# 1과목 1장: 전사아키텍처 개요

## 1. 전사아키텍처 정의

**전사아키텍처(Enterprise Architecture, EA)** 는 조직의 비즈니스 목표를 달성하기 위해 업무, 데이터, 애플리케이션, 기술 등 전사 구성 요소와 이들 간의 관계를 체계적으로 정의·관리하는 프레임워크이다.

| 구분 | 설명 |
|------|------|
| 목적 | 비즈니스-IT 정렬(Alignment), AS-IS → TO-BE 전환 계획 수립 |
| 범위 | 전사(Enterprise) 수준의 모든 IT 및 비즈니스 자원 |
| 핵심 산출물 | 아키텍처 뷰, 원칙, 표준, 로드맵 |

### EA의 4대 아키텍처 도메인

| 도메인 | 약어 | 주요 내용 | 산출물 예시 |
|--------|------|---------|----------|
| 비즈니스 아키텍처 | BA | 전략, 프로세스, 조직 | 업무 기능 분류체계, 업무 흐름도 |
| 데이터 아키텍처 | DA | 데이터 구조, 표준, 흐름 | 개념/논리 데이터 모델, 데이터 표준 |
| 애플리케이션 아키텍처 | AA | 응용 시스템 구조·관계 | 응용 기능 분류체계, 인터페이스 목록 |
| 기술 아키텍처 | TA | 하드웨어, 소프트웨어, 네트워크 | 기술 참조 모델, 시스템 구성도 |

> **핵심**: EA는 단순 IT 계획이 아닌 비즈니스-IT 정렬이 핵심 목표이다.

## 2. 전사아키텍처 프레임워크

EA 프레임워크는 아키텍처를 개발·관리하기 위한 체계적 방법론과 구조를 제공하는 틀이다.

### 2.1 Zachman Framework

존 재크만(John Zachman, 1987). **6 관점 × 6 역할 = 36개 셀** 구조.

**6가지 관점(Column):**

| 관점 | 키워드 | 아키텍처 요소 |
|------|--------|------------|
| What | 무엇을 | 데이터 |
| How | 어떻게 | 기능/프로세스 |
| Where | 어디서 | 네트워크/위치 |
| Who | 누가 | 사람/조직 |
| When | 언제 | 시간/이벤트 |
| Why | 왜 | 동기/목표/업무규칙 |

**6가지 역할(Row):**

| 행 | 역할 |
|----|------|
| 1 | 기획자(Planner)/범위 |
| 2 | 소유자(Owner)/기업 모델 |
| 3 | 설계자(Designer)/시스템 모델 |
| 4 | 구현자(Builder)/기술 모델 |
| 5 | 하도급자(Subcontractor)/세부 표현 |
| 6 | 운영자(User)/실제 시스템 |

### 2.2 TOGAF (The Open Group Architecture Framework)

The Open Group이 개발한 EA 방법론. **ADM(Architecture Development Method을 핵심으로 한다.

**ADM 순환 단계:**

```
예비 단계 → A(아키텍처 비전) → B(비즈니스 아키텍처) → C(정보시스템 아키텍처)
→ D(기술 아키텍처) → E(기회·해결방안) → F(이전계획) → G(구현거버넌스)
→ H(아키텍처 변경 관리) → (A로 순환)
```

| 단계 | 주요 산출물 |
|------|---------|
| A. 아키텍처 비전 | 이해관계자 분석, 비전 문서 |
| B. 비즈니스 아키텍처 | 비즈니스 프로세스 모델 |
| C. 정보시스템 아키텍처 | 데이터 모델, 애플리케이션 목록 |
| D. 기술 아키텍처 | 기술 표준, 플랫폼 명세 |
| E. 기회·해결방안 | 갭 분석, 후보 솔루션 |
| G. 구현 거버넌스 | 준수 여부 감독 |
| H. 변경 관리 | 변경 요청 처리 |

**TOGAF 아키텍처 리포지터리** 구성: 메타모델, 역량, 랜드스케이프, 표준 정보 기반

### 2.3 FEA (Federal Enterprise Architecture)

미국 연방정부 EA 체계. **5개 참조 모델** 구성:

| 약어 | 참조 모델 |
|------|---------|
| PRM | 성과 참조 모델 |
| BRM | 비즈니스 참조 모델 |
| SRM | 서비스 컴포넌트 참조 모델 |
| TRM | 기술 참조 모델 |
| DRM | 데이터 참조 모델 |

### 2.4 국내 ITA/EA 프레임워크

「전자정부법」에 근거한 국내 공공기관 EA 체계.

**6대 아키텍처 도메인** (국제 4대 + 국내 2개 추가):

| 도메인 | 비고 |
|--------|------|
| 비즈니스 아키텍처 | 국제 표준 |
| 데이터 아키텍처 | 국제 표준 |
| 애플리케이션 아키텍처 | 국제 표준 |
| 기술 아키텍처 | 국제 표준 |
| **보안 아키텍처** | 국내 추가 |
| **성과 아키텍처** | 국내 추가 |

## 3. 전사아키텍처 참조 모델

**EA 참조 모델(Reference Model)** 은 아키텍처 구성 요소를 표준화된 방식으로 분류·정의한 모델로, 기관 간 아키텍처 정보 공유와 재사용을 위한 공통 언어를 제공한다.

### 3.1 참조 모델의 역할

| 역할 | 설명 |
|------|------|
| 표준화 | 아키텍처 구성 요소의 공통 분류 체계 제공 |
| 재사용 | 기관 간 공통 서비스·컴포넌트 재사용 촉진 |
| 비교·평가 | AS-IS와 TO-BE 아키텍처 비교 기준 |
| 거버넌스 | 아키텍처 표준 준수 여부 판단 기준 |

### 3.2 국내 IT 참조 모델 (ITA)

| 참조 모델 | 내용 |
|---------|------|
| BRM (비즈니스 참조 모델) | 정부 업무 기능 분류체계 |
| SRM (서비스 참조 모델) | 공통 IT 서비스 분류 |
| TRM (기술 참조 모델) | 표준 기술 컴포넌트 분류 |
| DRM (데이터 참조 모델) | 데이터 항목·표준 분류 |

### 3.3 AS-IS / TO-BE 아키텍처

| 구분 | 설명 |
|------|------|
| AS-IS (현재 아키텍처) | 현재 운영 중인 IT 환경 |
| TO-BE (목표 아키텍처) | 미래 달성 목표 상태 |
| 전환 계획 (Transition Plan) | AS-IS → TO-BE 단계별 로드맵 |

## 4. 전사아키텍처 프로세스

EA 프로세스는 전사아키텍처를 수립·유지하는 단계적 활동 체계이다.

### 4.1 EA 개발 프로세스 단계

```
1. EA 준비 → 2. EA 방향 수립 → 3. EA 현황 파악(AS-IS)
→ 4. EA 목표 수립(TO-BE) → 5. 이행 계획 수립 → 6. EA 관리·유지
```

| 단계 | 주요 활동 |
|------|---------|
| EA 준비 | 조직·방법론 수립, 범위 정의 |
| 방향 수립 | 비전, 원칙, 목표 정의 |
| AS-IS 현황 파악 | 현행 아키텍처 정보 수집·문서화 |
| TO-BE 목표 수립 | 목표 아키텍처 설계 |
| 이행 계획 | 갭 분석, 프로젝트 로드맵 |
| 관리·유지 | 아키텍처 최신화, 거버넌스 운영 |

### 4.2 EA 성숙도 모델 (6단계)

| 수준 | 단계명 | 특징 |
|------|--------|------|
| 1 | 없음(None) | EA 인식 부재 |
| 2 | 시작(Initial) | 초보적 EA 수립 시도 |
| 3 | 개발 중(Under Development) | EA 개발 진행 |
| 4 | 완료(Completed) | EA 수립 완료 |
| 5 | 유지보수(Maintained) | 지속적 최신화 |
| 6 | 활용(Leveraged) | 의사결정에 적극 활용 |

### 4.3 EA 원칙 특성

| 특성 | 의미 |
|------|------|
| 보편성 | 전사 적용 가능한 일반 원칙 |
| 명확성 | 모호하지 않은 표현 |
| 안정성 | 자주 변경되지 않아야 함 |
| 완전성 | EA 전 영역 커버 |

## 5. 데이터 아키텍처(DA) 심화

### 5.1 아키텍처의 3가지 구성요소

| 구성요소 | 설명 |
|---------|------|
| Rule(규칙) | 조직의 전략·원칙·지침으로 아키텍처 설계의 기준 확립 |
| Plan(계획) | 현재와 목표 아키텍처를 정의하여 이행·구축 계획 수립 |
| Model(모델) | 참조모델/비즈니스/데이터/애플리케이션/기술 아키텍처 수립 |

### 5.2 효과적인 데이터 아키텍처의 특징

| 특징 | 설명 |
|------|------|
| 현행 비즈니스 지원 | 현행 비즈니스를 지원하면서 미래 변화에 유연하게 대응 |
| 핵심 데이터 집중 | 필수·핵심 데이터에 집중 (불필요한 모든 데이터 포함 지양) |
| Actionable(실행가능성) | 명확한 활동과 마일스톤 정의로 실행 가능해야 함 |
| Evolutionary(진화성) | 변화하는 요구사항과 기술에 대응 가능한 유연성 |

### 5.3 데이터 아키텍처 실패 요인

1. 불필요한 모든 데이터와 정보를 포함하려는 시도
2. 구조 설계에 대한 전문성 부족
3. IT 부서의 독점적 소유 (이해관계자 참여 필요)
4. 단계적 성과 미확인 (진행 중 검증 필요)

### 5.4 데이터 아키텍처 모델 계층

```
전사 데이터 영역 모델 → 개념 데이터 모델 → 논리 데이터 모델 → 물리 데이터 모델
     (주제 영역 분류)       (핵심 엔터티 도출)    (속성·관계 상세화)   (물리 DB 설계)
```

### 5.5 데이터 참조모델 구축 방법

1. 동종·유사 업종의 선진 참조모델 활용
2. 상위 및 연계 대상의 참조모델 활용
3. 외부(국외) 표준 활용
4. 현행 데이터 구조 참고
5. 외부 데이터 전문 인력을 통한 정의

### 5.6 데이터 참조모델 지속적 유지관리 기준

| 기준 | 설명 |
|------|------|
| 범용성 | 다양한 업무 영역에서 참조 가능 |
| 단순성 | 복잡하지 않은 단순한 모델 |
| 표준성 | 일반적인 표준 용어 사용 |
| 정확성 | 반드시 정확한 내용 |
| 정보 이용성 | 엔터티 정의·관계·규칙 등 포함 |
| 분류성 | 업무 영역과 업종별 분류 가능 |

### 5.7 데이터 참조모델 기대효과

1. 데이터 표준화 및 재사용 환경 구축
2. 이식성 및 확장성 향상
3. 조직 및 시스템 간 상호 운용성 향상
4. 데이터 품질 향상 및 생산성 증대
5. 중복 투자 방지 및 비용 절감

### 5.8 기술 참조모델(TRM)의 특징

1. 구축하기 쉽고 활용 효과가 커서 **가장 먼저 적용**되는 참조모델
2. 개별 기업에서도 기술 참조모델을 별도 정의하여 사용하는 것이 일반적
3. 기업에 적합한 DBMS 선택 기준 제공
4. 시스템 구축에 사용할 기술의 표준 프로파일 분류 기준 제공

## 출제 포인트

1. EA 4대 도메인: BA·DA·AA·TA (비데애기)
2. ITA 6대 도메인: 4대 + 보안·성과 아키텍처
3. 국내 ITA 법적 근거: 「전자정부법」
4. Zachman: 6×6 = 36개 셀, 분류 체계 중심
5. Zachman 6관점: What/How/Where/Who/When/Why
6. TOGAF: ADM 방법론, 예비+A~H 단계 순환
7. TOGAF ADM 단계 수: 예비 단계 포함 9단계
8. FEA 5대 참조 모델: PRM/BRM/SRM/TRM/DRM
9. EA 참조 모델 역할: 표준화·재사용·비교평가·거버넌스
10. EA 성숙도 6단계, 최고 = 활용(Leveraged)
11. EA 원칙 4특성: 보편성·명확성·안정성·완전성
12. AS-IS = 현재, TO-BE = 목표, 전환계획 = 로드맵
13. EA 프로세스: 준비→방향수립→AS-IS→TO-BE→이행계획→관리유지
