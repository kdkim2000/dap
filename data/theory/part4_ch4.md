# 4과목 4장: 물리 데이터 모델링

## 1. 물리 데이터 모델링 이해

**물리 데이터 모델링(Physical Data Modeling)**은 논리 데이터 모델을 특정 DBMS에서 구현하기 위해 테이블, 컬럼, 인덱스, 파티션 등 물리적 요소를 설계하는 단계이다.

| 특징 | 설명 |
|------|------|
| 관점 | 데이터베이스 구현 관점 |
| DBMS 종속성 | 특정 DBMS에 최적화 |
| 성능 고려 | 인덱스·파티션 등으로 성능 최적화 |
| 대상 독자 | DBA, 개발자 |
| 산출물 | 테이블 명세서, DDL 스크립트 |

---

## 2. 물리 요소 조사 및 분석

논리 모델을 물리 모델로 변환하기 전, 물리적 환경을 분석하는 단계이다.

### 2.0 물리 데이터 모델링 전 조사 항목

실제 모델링 수행 전 반드시 조사해야 하는 사항:

1. 시스템 구축 관련 **명명 규칙**
2. 운영체제 및 DBMS 버전
3. **하드웨어 자원**: CPU, 메모리, 디스크, I/O 컨트롤러, 네트워크
4. DBMS 파라미터 정보
5. 데이터베이스 운영 관련 관리 요소 (사용자 관리, 백업, 복구, 보안)

### 2.1 DBMS 특성 분석

| 분석 항목 | 내용 |
|---------|------|
| DBMS 종류 | Oracle, MySQL, SQL Server 등 |
| 지원 데이터 타입 | VARCHAR2, NUMBER 등 DBMS별 타입 |
| 최대 컬럼 수 | DBMS별 테이블 최대 컬럼 수 제한 |
| 인덱스 지원 | 클러스터드, 논클러스터드, 비트맵 등 |
| 파티셔닝 지원 | 범위, 해시, 리스트 파티션 등 |

### 2.2 데이터 볼륨 분석

| 분석 항목 | 내용 |
|---------|------|
| 테이블별 행 수 | 현재 및 예상 데이터 건수 |
| 연간 증가율 | 데이터 증가 속도 예측 |
| 데이터 크기 | 테이블별 예상 저장 용량 |
| 조회 패턴 | 자주 실행되는 쿼리 유형 및 빈도 |

### 2.3 성능 요건 분석

| 요건 | 내용 |
|------|------|
| 응답 시간 | 쿼리별 허용 응답 시간 목표 |
| 처리량 | 동시 처리 가능한 트랜잭션 수 |
| 가용성 | 시스템 가동 시간 목표(99.9% 등) |

## 3. 논리-물리 모델 변환

논리 ERD를 물리 테이블 구조로 변환하는 단계이다.

### 3.1 기본 변환 규칙

| 논리 모델 요소 | 물리 모델 요소 |
|------------|------------|
| 엔터티 | 테이블(Table) |
| 속성 | 컬럼(Column) |
| 기본 키(PK) | Primary Key 제약 |
| 보조 식별자(Alternate UID) | Unique Key 제약 |
| 외래 키(FK) | Foreign Key 제약 |
| 도메인 | 데이터 타입·길이·NOT NULL |
| 업무 제약(Business Constraints) | Check 제약 조건 |

### 3.2 테이블 설계 원칙

| 원칙 | 내용 |
|------|------|
| 컬럼 순서 | PK → 자주 사용하는 컬럼 → 나머지 |
| 이름 규칙 | 언더스코어 구분, 대문자 또는 소문자 일관성 |
| NOT NULL 최소화 | 필수 속성만 NOT NULL 지정 |

### 3.3 슈퍼타입/서브타입 변환 전략

| 전략 | 설명 | 장단점 |
|------|------|--------|
| One Table | 슈퍼+서브를 하나의 테이블 | 조회 간단, 데이터 희소성 |
| One Per Type | 슈퍼·서브 각각 별도 테이블 | 정규화 높음, JOIN 필요 |
| Concrete Table | 서브타입별 독립 테이블 (슈퍼타입 테이블 없음) | 유연, 데이터 중복 |

### 3.4 M:N 관계 변환

M:N 관계는 연결 테이블(교차 테이블)로 구현:

```sql
CREATE TABLE 수강 (
  학생ID   VARCHAR2(10) NOT NULL,
  강좌ID   VARCHAR2(10) NOT NULL,
  수강일자  DATE,
  PRIMARY KEY (학생ID, 강좌ID),
  FOREIGN KEY (학생ID) REFERENCES 학생(학생ID),
  FOREIGN KEY (강좌ID) REFERENCES 강좌(강좌ID)
);
```

### 3.5 무결성 제약 설계

| 제약 유형 | 설명 | SQL |
|---------|------|-----|
| 도메인 무결성 | 컬럼 값의 형식·범위 | CHECK 제약 |
| 개체 무결성 | PK는 NULL 불가·유일 | PRIMARY KEY |
| 참조 무결성 | FK 값은 부모 PK에 존재 | FOREIGN KEY |

## 4. 반정규화(De-normalization)

정규화로 분리된 구조를 의도적으로 합치거나 중복을 허용하여 **조회 성능을 향상**시키는 기법.

### 4.1 반정규화 적용 시점

| 기준 | 내용 |
|------|------|
| 조회 빈도 | 특정 JOIN이 매우 빈번하게 발생 |
| 성능 한계 | 정규화 상태에서 응답 시간 목표 미달성 |
| 데이터 안정성 | 참조 테이블 데이터가 거의 변경 안 됨 |

### 4.2 반정규화 기법

| 기법 | 설명 | 예시 |
|------|------|------|
| 테이블 병합 | 자주 JOIN되는 두 테이블을 하나로 합침 | 주문+주문상세 병합 |
| 컬럼 중복 | 자주 JOIN해서 가져오는 컬럼을 현재 테이블에 추가 | 주문 테이블에 고객명 추가 |
| 파생 컬럼 추가 | 집계 결과를 별도 컬럼으로 저장 | 주문 수량 합계 컬럼 |
| 테이블 수평 분할 | 행(Row) 기준으로 분할, 특정 범위 행에만 주로 접근하는 경우 | 연도별 거래 이력 분리 |
| 테이블 수직 분할 | 컬럼(Column) 기준으로 분할, 자주/드물게 접근하는 컬럼 분리 | 기본정보·상세정보 분리 |
| 중복 테이블 생성 | 여러 테이블에서 필요 데이터를 주기적으로 추출하여 별도 테이블 생성 | 집계 테이블, 진행 테이블 |

**중복 테이블 유형:**

| 유형 | 설명 |
|------|------|
| **집계 테이블** | 통계·요약 자료 제공용 (SUM, COUNT 등 집계 결과 저장) |
| **진행 테이블** | 업무 진행 현황 요약 제공용 |

> **주의**: 중복 테이블은 인덱스 조정이나 클러스터링으로 해결 불가할 때 **최후의 방법**으로 사용

### 4.3 반정규화의 문제점

| 문제점 | 설명 |
|--------|------|
| 데이터 중복 | 동일 데이터가 여러 테이블에 존재 |
| 무결성 관리 어려움 | 중복 데이터 동기화 필요 |
| 수정·삭제 이상 가능성 | 반정규화 적용 시 이상 현상 재발 가능 |

## 5. 물리 데이터 모델 품질 검토

### 5.1 인덱스(Index) 설계

| 구분 | 설명 |
|------|------|
| 클러스터드 인덱스 | 데이터가 인덱스 순서로 물리적 정렬 (테이블당 1개) |
| 논클러스터드 인덱스 | 별도 인덱스 구조로 데이터 위치 참조 (여러 개 가능) |
| 복합 인덱스 | 두 개 이상 컬럼으로 구성된 인덱스 |

**인덱스 설계 원칙:**

| 원칙 | 내용 |
|------|------|
| 분포도 높은 컬럼 우선 | 중복이 적은(선택성 높은) 컬럼 |
| WHERE 절 조건 컬럼 | 자주 검색 조건으로 사용되는 컬럼 |
| JOIN 컬럼 | 자주 JOIN에 사용되는 컬럼 |
| DML 부하 고려 | 인덱스가 많을수록 DML 성능 저하 |

### 5.2 파티셔닝(Partitioning) 설계

대용량 테이블을 관리 용이성과 성능 향상을 위해 물리적으로 분할.

| 파티션 유형 | 기준 | 예시 |
|---------|------|------|
| 범위(Range) | 값의 범위 | 날짜별 거래 이력 |
| 해시(Hash) | 해시 함수 결과 | 균등 분산 |
| 리스트(List) | 특정 값 목록 | 지역코드별 |
| 복합(Composite) | 두 기준 조합 | 날짜+해시 |

**파티션 프루닝(Partition Pruning)**: 쿼리 조건이 파티션 키와 일치할 때 해당 파티션만 스캔하여 성능 향상.

### 5.3 물리 모델 품질 검토 항목

| 검토 항목 | 내용 |
|---------|------|
| 테이블명·컬럼명 | 명명 규칙 준수 |
| 데이터 타입 | 업무 요건·도메인과 일치 |
| 인덱스 설계 | 조회 성능 요건 충족 |
| 반정규화 영향 | 무결성 관리 방안 마련 |
| 파티셔닝 적용 | 대용량 테이블 적절한 파티셔닝 |
| DDL 문법 | DBMS 문법·제약 준수 |

## 출제 포인트

1. 물리 데이터 모델링: DBMS 종속적, 성능 최적화 목적
2. 논리→물리 변환: 엔터티→테이블, 속성→컬럼, PK/FK→제약
3. One Table: 슈퍼+서브 단일 테이블, 조회 간단하나 데이터 희소성
4. One Per Type: 슈퍼·서브 별도 테이블, 정규화 높음, JOIN 필요
5. 반정규화 목적: 조회 성능 향상 (JOIN 감소)
6. 반정규화 기법: 테이블 병합·컬럼 중복·파생 컬럼·테이블 분할
7. 반정규화 문제점: 데이터 중복·무결성 관리·이상 현상 가능성
8. 클러스터드 인덱스: 데이터 물리적 정렬, 테이블당 1개
9. 인덱스 설계: 분포도 높은 컬럼·WHERE 조건·JOIN 컬럼 우선
10. 파티션 프루닝: 쿼리 조건과 파티션 키 일치 시 해당 파티션만 스캔
11. 파티션 유형: 범위·해시·리스트·복합 파티션
12. 물리 모델 품질 검토: 명명규칙·데이터타입·인덱스·반정규화·파티션·DDL
