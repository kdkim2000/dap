# 6과목 2장: 데이터 구조 이해

## 1. 개념 데이터 모델

**개념 데이터 모델**은 업무 관점에서 핵심 엔터티와 관계를 추상적으로 표현한 모델이다. 기술적 세부사항 없이 업무 이해관계자와 소통하기 위해 사용한다.

### 1.1 개념 데이터 모델 구성 요소

| 구성 요소 | 설명 | 예시 |
|------|------|------|
| 주제 영역(Subject Area) | 관련 엔터티의 논리적 그룹 | 고객관리, 주문관리 |
| 핵심 엔터티(Core Entity) | 업무에서 중요한 핵심 개념 | 고객, 주문, 상품 |
| 관계(Relationship) | 엔터티 간의 업무적 연관성 | 고객이 주문을 한다 |

### 1.2 데이터 모델 3계층 비교

| 구분 | 개념 모델 | 논리 모델 | 물리 모델 |
|------|------|------|------|
| 대상 독자 | 업무 전문가, 이해관계자 | 데이터 전문가 | DBA, 개발자 |
| 표현 수준 | 추상적 (핵심 엔터티만) | 상세 (속성, 식별자 포함) | DBMS 최적화 |
| 정규화 | 미적용 | 적용 (3NF) | 반정규화 가능 |
| DBMS 독립성 | 완전 독립 | 완전 독립 | DBMS 종속 |
| 주요 산출물 | 핵심 엔터티 목록, 주제 영역 | 엔터티·속성·관계 정의서 | DDL, 테이블 정의서 |

## 2. 데이터 참조 모델

**데이터 참조 모델(Data Reference Model)** 은 표준화된 방식으로 데이터를 분류·정의한 모델로, 조직 간 데이터 공유와 재사용을 지원한다.

### 2.1 참조 모델의 역할

- 데이터 표준화의 기준 제공
- 기관 간 데이터 상호 운용성 향상
- 중복 데이터 모델 개발 방지
- AS-IS와 TO-BE 비교 기준

### 2.2 국내 ITA 데이터 참조 모델(DRM)

공공기관의 데이터 표준화를 위한 참조 모델.
- 업무 데이터 분류 체계 제공
- 표준 용어·코드 정의
- 데이터 공유 기반 마련

## 3. 논리 데이터 모델

**논리 데이터 모델**은 업무 요구사항을 상세하게 표현한 데이터 모델로, DBMS에 독립적이며 정규화가 적용된다.

### 3.1 논리 데이터 모델 구성 요소

| 구성 요소 | 설명 |
|------|------|
| 엔터티(Entity) | 관리 대상이 되는 개념·사건·장소 |
| 속성(Attribute) | 엔터티의 특성이나 성질 |
| 식별자(Identifier) | 엔터티 인스턴스를 유일하게 식별하는 속성 집합 |
| 관계(Relationship) | 두 엔터티 간의 업무적 연관성 |
| 서브타입(Subtype) | 엔터티의 하위 유형 (일반화·특수화) |

### 3.2 엔터티 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| 기본 엔터티 | 업무의 핵심 개념 | 고객, 상품, 직원 |
| 중심 엔터티 | 기본 엔터티 간의 행위 | 주문, 계약 |
| 행위 엔터티 | 업무 행위의 이력 | 주문이력, 변경이력 |

### 3.3 관계 표기법

**바커(Barker) 표기법**:
- 실선: 필수 관계 (Mandatory)
- 점선: 선택 관계 (Optional)
- 까마귀발 기호: 다(多) 표시

**IE(Information Engineering) 표기법**:
- 실선·점선으로 필수·선택 구분
- 단선·복선으로 1·다(多) 구분

> **핵심**: 바커 표기법과 IE 표기법을 혼용하면 안 된다. 하나만 일관되게 사용.

## 4. 물리 데이터 모델

**물리 데이터 모델**은 논리 데이터 모델을 특정 DBMS에 맞게 변환한 모델이다.

### 4.1 논리 → 물리 변환

| 논리 모델 요소 | 물리 모델 요소 |
|------|------|
| 엔터티 | 테이블(TABLE) |
| 속성 | 컬럼(COLUMN) |
| 식별자 | 기본키(PRIMARY KEY) |
| 관계 | 외래키(FOREIGN KEY) |
| 서브타입 | 테이블 통합/분리 또는 슈퍼타입+서브타입 |

### 4.2 반정규화(De-normalization)

성능 향상을 위해 의도적으로 정규화 규칙을 위반하는 기법.

| 기법 | 설명 |
|------|------|
| 테이블 합병 | 자주 조인되는 테이블을 하나로 합침 |
| 컬럼 중복 | 자주 조회하는 컬럼을 다른 테이블에 복사 |
| 파생 컬럼 추가 | 계산 결과를 컬럼으로 저장 (예: 총합 컬럼) |
| 테이블 분할 | 컬럼 수가 많은 테이블을 분리 |

> **주의**: 반정규화는 데이터 중복을 허용하므로 일관성 관리가 중요하다.

### 4.3 서브타입 물리 구현 전략

논리 모델의 슈퍼타입/서브타입을 물리 모델로 변환하는 3가지 전략.

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **통합(Single Table)** | 슈퍼+서브 속성을 하나의 테이블에 | 조인 불필요 | NULL 컬럼 다수 발생 |
| **분리(Separate Tables)** | 서브타입별 독립 테이블 | NULL 없음 | 조인 필요 |
| **혼합(Mixed)** | 슈퍼타입 + 서브타입별 테이블 | 균형적 | 설계 복잡 |

> **선택 기준**: 서브타입 간 속성 차이가 크면 분리, 공통 조회가 많으면 통합 전략 고려.

### 4.4 파티셔닝 (Partitioning)

대용량 테이블을 기준에 따라 물리적으로 분할하여 관리·성능을 향상시키는 기법.

| 유형 | 분할 기준 | 설명 | 사용 예 |
|------|------|------|------|
| **Range 파티셔닝** | 컬럼 값의 연속적 범위 | 날짜·숫자 범위로 분할 | 연월별 판매 데이터 |
| **List 파티셔닝** | 컬럼 값의 목록 | 특정 값 목록으로 분할 | 지역코드(서울/부산/대구) |
| **Hash 파티셔닝** | 해시 함수 적용 결과 | 균등 분산 보장 | 균형적 분산이 필요할 때 |
| **Composite 파티셔닝** | 두 가지 이상 기준 복합 | Range+Hash 등 조합 | 연월+지역 2차원 분할 |

```sql
-- Range 파티셔닝 예시
CREATE TABLE TB_SALES (
  sale_id NUMBER, sale_date DATE, amount NUMBER
) PARTITION BY RANGE(sale_date) (
  PARTITION p_2024_q1 VALUES LESS THAN (DATE '2024-04-01'),
  PARTITION p_2024_q2 VALUES LESS THAN (DATE '2024-07-01'),
  PARTITION p_2024_q3 VALUES LESS THAN (DATE '2024-10-01'),
  PARTITION p_2024_q4 VALUES LESS THAN (DATE '2025-01-01')
);
```

**파티셔닝 효과**:
- **파티션 프루닝**: WHERE 조건이 파티션 키이면 해당 파티션만 스캔 → I/O 대폭 감소
- **병렬 처리**: 파티션별 병렬 DML 가능
- **관리 가용성**: 파티션 단위 백업·복구·삭제 가능

### 4.5 재귀 관계 (Recursive Relationship)

동일 엔터티 내의 인스턴스들이 서로 관계를 갖는 구조. 자기 참조 관계라고도 한다.

```
[직원]
 - 사번(PK)
 - 이름
 - 상위사번(FK → 사번)  ← 자기 자신을 참조
```

**물리 구현**:

```sql
CREATE TABLE TB_EMPLOYEE (
  emp_id    NUMBER PRIMARY KEY,
  emp_name  VARCHAR2(100),
  mgr_id    NUMBER REFERENCES TB_EMPLOYEE(emp_id)  -- 자기 참조 FK
);

-- 계층 조회 (Oracle)
SELECT emp_id, emp_name, mgr_id,
       LEVEL, LPAD(' ', (LEVEL-1)*2) || emp_name AS hierarchy
FROM TB_EMPLOYEE
START WITH mgr_id IS NULL      -- 최상위 (CEO)
CONNECT BY PRIOR emp_id = mgr_id;  -- 계층 연결
```

**사용 예**:
- 조직도 (직원-상사 관계)
- 카테고리 트리 (대분류-중분류-소분류)
- BOM(Bill of Materials, 제품-부품 구성)

## 5. 데이터베이스

**데이터베이스**는 물리 데이터 모델이 실제로 구현된 저장소이다.

### 5.1 데이터 딕셔너리(Data Dictionary)

DBMS가 관리하는 메타데이터 저장소.
- 테이블, 컬럼, 인덱스, 권한 정보 포함
- 시스템 카탈로그(System Catalog)라고도 함

### 5.2 데이터베이스 스키마 계층

| 계층 | 설명 |
|------|------|
| 외부 스키마(External) | 사용자 뷰 수준 (사용자별 다른 뷰) |
| 개념 스키마(Conceptual) | 논리적 데이터 전체 구조 |
| 내부 스키마(Internal) | 물리적 저장 구조 |

## 6. 사용자 뷰(User View)

**뷰(View)** 는 하나 이상의 테이블을 기반으로 만든 가상 테이블이다.

### 6.1 뷰의 특징

| 특징 | 설명 |
|------|------|
| 독립성 | 테이블 구조 변경 시 뷰 정의만 수정하면 됨 |
| 보안성 | 민감 컬럼을 제외하고 접근 허용 |
| 편의성 | 복잡한 쿼리를 단순화 |
| 가상성 | 물리적 데이터를 보유하지 않음 (마테리얼라이즈드 뷰 제외) |

### 6.2 마테리얼라이즈드 뷰 (Materialized View)

일반 뷰와 달리 **쿼리 결과를 물리적으로 저장**하는 특수 뷰. 조회 시 계산 없이 저장된 결과를 즉시 반환.

| 구분 | 일반 뷰(View) | 마테리얼라이즈드 뷰(Materialized View) |
|------|------|------|
| 데이터 저장 | 저장 안 함 (가상) | **물리적으로 저장** |
| 조회 성능 | 매번 원본 쿼리 실행 | 저장된 결과 즉시 반환 → 빠름 |
| 최신성 | 항상 최신 | 갱신 주기에 따라 지연 가능 |
| 갱신 방법 | 해당 없음 | 완전 갱신 또는 증분 갱신 |
| 주요 용도 | 보안·편의성 | DW 집계 성능 향상 |

```sql
-- 마테리얼라이즈드 뷰 생성 (Oracle)
CREATE MATERIALIZED VIEW MV_SALES_MONTHLY
BUILD IMMEDIATE               -- 즉시 생성
REFRESH COMPLETE ON DEMAND    -- 요청 시 완전 갱신
AS
SELECT TRUNC(sale_date, 'MM') AS sale_month,
       SUM(amount) AS total_amount,
       COUNT(*) AS sale_count
FROM TB_SALES
GROUP BY TRUNC(sale_date, 'MM');
```

> **핵심**: DW 환경에서 집계 쿼리가 느릴 때 마테리얼라이즈드 뷰로 사전 계산하여 성능 향상. 갱신 주기를 적절히 설정해야 최신성 보장.

## 7. 정규화(Normalization)

**정규화**는 데이터 중복과 이상(Anomaly)을 제거하여 데이터 일관성을 보장하는 과정이다.

### 7.1 함수 종속성(Functional Dependency)

`A → B`: A가 B를 결정한다 (A가 같으면 B도 같다).
- 완전 함수 종속: 기본키 전체에 종속
- 부분 함수 종속: 기본키 일부에 종속 (복합키일 때 발생)
- 이행 함수 종속: A→B, B→C이면 A→C

### 7.2 정규화 단계

**제1정규형(1NF)** — 원자값(Atomic Value)만 허용

| 위반 예 | 수강과목이 "수학,영어,과학" 한 컬럼에 저장 |
|------|------|
| 해결 | 수강과목을 별도 행으로 분리 (반복 그룹 제거) |

**제2정규형(2NF)** — 1NF + 부분 함수 종속 제거

```
[위반] PK: (학번, 과목코드)
  - 성적: 학번+과목코드에 종속 ✓
  - 학생명: 학번에만 종속 ← 부분 종속!
[해결] 학생 테이블(학번, 학생명) + 수강 테이블(학번, 과목코드, 성적)
```

**제3정규형(3NF)** — 2NF + 이행 함수 종속 제거

```
[위반] 학번 → 학과코드 → 학과장명
  - 학과장명이 학번에 이행 종속!
[해결] 학생(학번, 학과코드) + 학과(학과코드, 학과장명)
```

**BCNF(보이스-코드 정규형)** — 3NF + 모든 결정자가 후보키

```
[위반] (학번, 과목) → 교수, 교수 → 과목
  - 교수가 결정자이지만 후보키가 아님!
[해결] 교수(교수, 과목) + 수강(학번, 교수)
```

### 7.3 이상(Anomaly) 현상

| 이상 유형 | 설명 |
|------|------|
| 삽입 이상 | 일부 데이터 삽입 시 불필요한 데이터도 삽입해야 함 |
| 삭제 이상 | 특정 데이터 삭제 시 의도치 않은 데이터도 삭제됨 |
| 갱신 이상 | 중복 데이터 수정 시 일부만 수정되어 불일치 발생 |

> **핵심**: DAP 실기에서 논리 데이터 모델은 반드시 최소 3차 정규화(3NF) 적용 확인 필요.

## 출제 포인트

1. **데이터 모델 3계층**: 개념 → 논리 → 물리
2. **3계층 비교**: 개념(업무전문가/추상) → 논리(정규화/DBMS독립) → 물리(DBMS종속/반정규화)
3. **엔터티 3유형**: 기본·중심·행위 엔터티
4. **바커 vs IE 표기법**: 혼용 금지, 하나만 사용
5. **논리→물리 변환**: 엔터티→테이블, 속성→컬럼, 관계→외래키
6. **정규화 3단계**: 1NF(원자값)→2NF(부분종속제거)→3NF(이행종속제거)
7. **BCNF**: 모든 결정자가 후보키여야 함 (3NF보다 강한 조건)
8. **이행 함수 종속**: A→B, B→C → A→C (3NF에서 제거 대상)
9. **반정규화**: 성능을 위해 의도적으로 정규화 위반
10. **서브타입 구현 전략**: 통합(NULL 많음) / 분리(조인 필요) / 혼합
11. **파티셔닝 4가지**: Range(범위)·List(목록)·Hash(해시)·Composite(복합)
12. **파티션 프루닝**: WHERE 조건이 파티션 키이면 해당 파티션만 스캔 → I/O 감소
13. **재귀 관계(Recursive Relationship)**: 자기 참조 FK — 계층 구조(조직도·카테고리·BOM)
14. **CONNECT BY**: Oracle 계층 쿼리 키워드 (START WITH: 최상위, PRIOR: 부모 연결)
15. **마테리얼라이즈드 뷰**: 쿼리 결과를 물리 저장 → 조회 성능↑, DW 집계에 활용
16. **일반 뷰 vs 마테리얼라이즈드 뷰**: 가상(항상 최신) vs 물리 저장(갱신 주기 지연 가능)
17. **뷰(View)**: 가상 테이블, 보안·편의성 제공
18. **데이터 딕셔너리**: DBMS가 관리하는 메타데이터 저장소
19. **3계층 스키마**: 외부(뷰)→개념(논리)→내부(물리)
20. **참조 모델**: 기관 간 데이터 공유·표준화 기준
