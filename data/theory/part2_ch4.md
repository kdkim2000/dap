# 2과목 4장: 정보 요구 검증

## 0. 정보 요구사항 생명주기와 명세화

### 0.1 정보 요구사항 생명주기

```
도출 → 분석/정의 → 명세화 → 검증 → 승인
  ↑                              ↓
  └──── 추적·변경 요청·영향 분석·승인/기각 ────┘
```

### 0.2 정보 요구 명세서의 역할

| 역할 주체 | 활용 목적 |
|---------|---------|
| **사용자** | 개발 완료 판단의 기준 |
| **개발자** | 분석·설계·코딩의 기초 문서 |
| **테스터** | 테스트 케이스 작성 기준 |

> **핵심**: 정보 요구 명세서는 "사용자와 개발자를 연결하는 문서"이다.

### 0.3 명세서 작성 시 주의사항

1. 사용자 이해도 중심의 명확한 작성
2. 개발 가능성을 고려한 기술
3. **정량적** 테스트 기준 제시
4. 비기능 요구(성능·보안·가용성) 명시
5. 품질 우선순위 명확화

### 0.4 요구사항 검증 3가지 방식

| 검증 방식 | 설명 |
|---------|------|
| **명세서 검토** | 일관성과 적절성 검토 |
| **용어 검증** | 일관성·표준성·이해 용이성 만족 여부 검토 |
| **베이스라인 설정** | 검토 완료 후 공식 기준선 확립 |

### 0.5 상관분석 수행 주체 비교

| 수행 주체 | 장점 | 단점 |
|---------|------|------|
| **요구사항 분석가** | 업무 이해도 높음 | 객관성 저하 우려 |
| **품질보증팀(QA)** | 인터페이스 검증 용이 | 업무 이해도 낮음 |
| **외부 감리** | 제3자 객관성 극대화 | 업무 파악 한계 |

## 1. 정보 요구 사항 상관분석 기법

정보 요구 사항 상관분석은 여러 정보 요구 사항 간의 관계·연관성·충돌을 체계적으로 분석하는 기법이다.

### 1.1 상관분석의 목적

| 목적 | 내용 |
|------|------|
| 충돌 식별 | 서로 모순되거나 충돌하는 요구 사항 발견 |
| 의존성 파악 | 요구 사항 간 선후 관계·의존 관계 파악 |
| 중복 탐지 | 유사하거나 동일한 요구 사항 중복 확인 |
| 누락 발견 | 다른 요구 사항에서 유추 가능한 누락 요건 도출 |

### 1.2 상관 행렬(Correlation Matrix) 기법

요구 사항들 간의 관계를 행렬 형태로 표현하는 기법.

| 관계 유형 | 기호 | 의미 |
|---------|------|------|
| 강한 상관 | ◎ | 두 요구 사항이 강하게 연관 |
| 약한 상관 | ○ | 두 요구 사항이 약하게 연관 |
| 부정적 상관 | × | 두 요구 사항이 충돌 |
| 무관 | (공백) | 관계 없음 |

**활용 예시:**
```
요구IR | IR-001 | IR-002 | IR-003
IR-001 |   —    |   ◎   |   ×
IR-002 |   ◎   |   —    |   ○
IR-003 |   ×   |   ○   |   —
```

### 1.3 의존성 분석

요구 사항 간의 선행·후행 관계를 파악하는 분석.

| 의존 유형 | 설명 |
|---------|------|
| 기능적 의존 | A 요구 구현 시 B 요구도 반드시 구현 필요 |
| 데이터 의존 | A 요구가 사용하는 데이터를 B 요구에서 생성 |
| 비즈니스 의존 | 업무 흐름상 A 선행 후 B 처리 |

### 1.3a CRUD 상관분석 기법

**정보요구/애플리케이션 상관분석:**
- CRUD 액션(Create·Read·Update·Delete) 기반으로 각 프로세스별 요구사항 반영 여부 확인
- 주의사항:
  - 정보항목과 프로세스 누락 방지
  - 상태 종료 프로세스 반드시 포함
  - 항목당 액션 7개 이하 권장

**정보요구/업무(조직) 기능 상관분석:**
- Create·Change·Use 기준으로 업무 기능별 요구사항 반영도 확인
- 비연관 업무는 재검토

### 1.4 연관분석(Association Analysis)

유사 업무나 데이터를 공유하는 요구 사항 그룹을 식별하는 분석.

- 같은 데이터를 사용하는 요구 사항 그룹화
- 같은 주제 영역에 속하는 요구 사항 연계
- 공통 데이터 표준 적용 대상 식별

## 2. 추가 및 삭제 정보 요구 사항 도출

검증 과정에서 새롭게 추가해야 할 요구 사항과 제거해야 할 불필요한 요구 사항을 도출하는 단계이다.

### 2.1 추가 정보 요구 사항 도출

상관분석 결과, 기존 요구 사항에서 암묵적으로 필요하지만 명시되지 않은 요구 사항을 발굴한다.

| 도출 방법 | 내용 |
|---------|------|
| 상관분석 결과 활용 | 상관 관계에서 유추 가능한 누락 요구 사항 |
| 업무 흐름 검토 | 업무 흐름도 검토 시 발견되는 누락 단계 |
| 법규·컴플라이언스 검토 | 법적 요건에서 도출되는 추가 요구 사항 |
| 데이터 품질 요건 보완 | 주요 데이터 항목의 품질 요건 누락 확인 |

**추가 요구 사항 발굴 시 확인 질문:**
- "이 데이터는 어디서 생성되는가?"
- "이 프로세스의 예외 처리는 어떻게 되는가?"
- "해당 데이터의 보존 기간은 얼마인가?"
- "이 데이터에 접근할 수 있는 권한은 누구에게 있는가?"

### 2.2 삭제 정보 요구 사항 도출

중복, 범위 외, 비현실적인 요구 사항을 식별하여 제거하는 활동.

| 삭제 기준 | 내용 |
|---------|------|
| 중복 요구 사항 | 통합 과정에서 누락된 중복 요구 사항 |
| 범위 외 요구 사항 | 프로젝트 범위 밖의 요구 사항 |
| 비실현 가능 요구 사항 | 기술적·비용적으로 구현 불가한 요구 사항 |
| 이미 구현된 기능 | 현행 시스템에 이미 존재하는 기능 |

### 2.3 요구 사항 변경 관리

추가·삭제 결정 시 공식적인 변경 관리 절차를 따른다.

```
변경 요청 제출 → 영향 분석 → 이해관계자 검토·승인 → 요구 명세서 업데이트 → 추적성 매트릭스 업데이트
```

| 관리 항목 | 내용 |
|---------|------|
| 변경 이력 | 언제, 누가, 무엇을 변경했는지 기록 |
| 영향 분석 | 변경 시 다른 요구 사항에 미치는 영향 분석 |
| 승인 이력 | 이해관계자 승인 기록 |

## 3. 정보 요구 보완 및 확인

검증 결과를 토대로 정보 요구 사항을 최종 보완하고 이해관계자의 최종 확인을 받는 단계이다.

### 3.1 보완 활동

| 활동 | 내용 |
|------|------|
| 누락 요구 사항 추가 | 상관분석·검토에서 발견된 누락 요구 추가 |
| 불명확한 요구 수정 | 모호한 표현을 구체적으로 수정 |
| 충돌 요구 조정 | 상충하는 요구 사항 중재·조정 |
| 우선순위 재조정 | 변경된 요구 사항 반영하여 우선순위 재설정 |

### 3.2 최종 확인 절차

| 단계 | 내용 |
|------|------|
| 검토 회의 준비 | 보완된 정보 요구 사항 명세서 배포 |
| 이해관계자 검토 | 현업·IT·관리자 등 검토 |
| 의견 수렴 | 이견 조정 및 합의 도출 |
| 최종 승인 | 이해관계자 서명·승인 |
| 기준선(Baseline) 설정 | 승인된 요구 사항을 기준선으로 확정 |

### 3.3 기준선(Baseline) 관리

**기준선(Configuration Baseline)** 은 공식적으로 검토·승인된 정보 요구 사항의 확정 버전이다.

| 특성 | 설명 |
|------|------|
| 변경 통제 대상 | 기준선 변경 시 공식 변경 요청 필요 |
| 추적성 기준 | 이후 설계·구현의 참조 기준 |
| 버전 관리 | 버전 번호로 변경 이력 관리 |

### 3.4 최종 산출물

| 산출물 | 내용 |
|--------|------|
| 확정 정보 요구 사항 명세서 | 승인된 최종 요구 사항 문서 |
| 요구 사항 추적성 매트릭스(RTM) | 요구 사항 ↔ 설계 ↔ 구현 추적 매핑 |
| 변경 요청 목록 | 추가·삭제 변경 이력 |

### 3.5 요구 사항 추적성 매트릭스(RTM: Requirements Traceability Matrix)

| 요구 ID | 요구 명칭 | 설계 산출물 | 구현 모듈 | 테스트 케이스 |
|--------|---------|---------|---------|------------|
| IR-001 | 고객 정보 관리 | ERD: 고객 엔터티 | 고객관리 모듈 | TC-001 |
| IR-002 | 주문 처리 | ERD: 주문 엔터티 | 주문관리 모듈 | TC-002 |

## 출제 포인트

1. 상관 행렬: ◎(강한 상관), ○(약한 상관), ×(충돌) 기호로 요구 사항 간 관계 표현
2. 상관분석 목적: 충돌 식별·의존성 파악·중복 탐지·누락 발견
3. 의존성 유형: 기능적·데이터·비즈니스 의존
4. 추가 요구 도출 방법: 상관분석 결과·업무흐름·법규·데이터 품질 검토
5. 삭제 기준: 중복·범위 외·비실현 가능·이미 구현된 기능
6. 변경 관리 절차: 요청→영향분석→승인→명세서 업데이트→추적성 업데이트
7. 기준선(Baseline): 공식 승인된 요구 사항의 확정 버전, 이후 설계의 참조 기준
8. RTM(요구 사항 추적성 매트릭스): 요구 ↔ 설계 ↔ 구현 ↔ 테스트 연결 추적
9. 최종 확인: 검토 회의→이해관계자 검토→합의→승인→기준선 설정
10. 보완 활동: 누락 추가·불명확 수정·충돌 조정·우선순위 재조정
