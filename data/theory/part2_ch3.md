# 2과목 3장: 정보 요구 사항 분석

## 1. 분석 대상 정의

분석 대상 정의는 통합된 정보 요구 사항 중 실제 데이터 모델링·시스템 개발 대상이 될 요구 사항의 범위를 결정하는 단계이다.

### 1.1 분석 대상 선정 기준

| 기준 | 내용 |
|------|------|
| 비즈니스 가치 | 업무 목표 달성에 핵심적인 정보 |
| 우선순위 | MoSCoW 기법으로 결정된 Must Have 중심 |
| 실현 가능성 | 기술적·예산적으로 구현 가능한 범위 |
| 데이터 주제 영역 | EA 데이터 아키텍처의 주제 영역과 연계 |

### 1.2 분석 제외 기준

| 제외 기준 | 이유 |
|---------|------|
| 중복 요구 사항 | 통합 단계에서 이미 처리됨 |
| 시스템 외부 영역 | 프로젝트 범위 밖의 요구 사항 |
| 현행 유지 영역 | 변경이 필요 없는 현행 기능 |

### 1.3 분석 대상 데이터 영역 분류

정보 요구 사항을 데이터 관점에서 분류:

| 분류 | 설명 |
|------|------|
| 마스터 데이터 | 변경이 적은 기준 정보 (고객, 상품, 조직) |
| 트랜잭션 데이터 | 업무 처리 결과 데이터 (주문, 결제, 배송) |
| 이력 데이터 | 시간에 따른 변경 이력 관리 데이터 |
| 참조 데이터 | 코드, 분류 체계 등 참조용 데이터 |

## 2. 정보 요구 사항 상세화

분석 대상으로 확정된 정보 요구 사항을 구체적이고 측정 가능하게 상세히 기술하는 단계이다.

### 2.1 상세화 활동

| 활동 | 내용 |
|------|------|
| 데이터 항목 도출 | 요구 사항에서 필요한 데이터 항목(속성) 목록화 |
| 데이터 관계 정의 | 데이터 항목·엔터티 간의 관계 파악 |
| 업무 규칙 명세 | 데이터 생성·변경·삭제에 관한 규칙 정의 |
| 데이터 품질 요건 | 정확성·완전성·일관성·적시성 요건 정의 |

### 2.2 업무 규칙 명세

업무 규칙(Business Rule)은 정보 요구 사항 상세화의 핵심 활동이다.

| 규칙 유형 | 설명 | 예시 |
|---------|------|------|
| 정의 규칙 | 용어의 의미·범위 정의 | "고객 = 1회 이상 구매한 자" |
| 사실 규칙 | 개념 간 구조·관계 기술 | "주문은 하나 이상의 주문항목 포함" |
| 제약 규칙 | 데이터·프로세스 제약 조건 | "잔액 >= 0", "주문 시 고객 존재 필수" |
| 파생 규칙 | 기존 데이터에서 새 값 계산 | "세금 = 금액 × 세율" |

### 2.3 데이터 흐름 분석

데이터가 시스템 내에서 어떻게 생성·처리·출력되는지 분석.

**DFD(Data Flow Diagram) 구성 요소:**

| 기호 | 의미 |
|------|------|
| 프로세스(원) | 데이터 처리 활동 |
| 데이터 저장소(직사각형) | 데이터 저장 위치 |
| 외부 개체(사각형) | 데이터 입출력 주체 |
| 데이터 흐름(화살표) | 데이터 이동 방향 |

### 2.4 상세화 산출물

| 산출물 | 내용 |
|--------|------|
| 상세 정보 요구 사항 명세서 | 데이터 항목·규칙·품질 요건 포함 |
| 개념 데이터 모델(초안) | 주요 엔터티와 관계를 도식화한 초안 ERD |
| 업무 규칙 목록 | 유형별 업무 규칙 목록 |
| 데이터 흐름도(DFD) | 데이터 처리 흐름 시각화 |

## 3. 정보 요구 사항 확인

상세화된 정보 요구 사항을 이해관계자와 함께 검토·확인하는 단계이다.

### 3.1 확인 목적

| 목적 | 내용 |
|------|------|
| 정확성 검증 | 이해관계자의 실제 요구와 일치하는지 확인 |
| 완전성 검증 | 누락된 요구 사항이 없는지 확인 |
| 일관성 검증 | 요구 사항 간 충돌·모순이 없는지 확인 |
| 실현가능성 검증 | 기술적으로 구현 가능한지 확인 |

### 3.2 확인 기법

| 기법 | 설명 |
|------|------|
| 인스펙션(Inspection) | 독립적 검토자가 사전 검토 후 공식 회의에서 결함 식별 |
| 워크스루(Walkthrough) | 작성자가 검토자에게 내용을 설명하며 함께 검토 |
| 프로토타입 검토 | 시각적 프로토타입을 통해 이해관계자 피드백 수집 |
| 체크리스트 검토 | 사전 정의된 점검 항목으로 체계적 검토 |

### 3.3 인스펙션 vs 워크스루 비교

| 구분 | 인스펙션 | 워크스루 |
|------|---------|---------|
| 형식 | 공식적 | 비공식적 |
| 검토자 | 독립적 전문가 | 팀 내 검토자 |
| 사전 준비 | 검토자가 사전 검토 | 상대적으로 간단 |
| 목적 | 결함 탐지에 집중 | 이해 공유 및 검토 |

### 3.4 확인 결과 처리

| 상태 | 처리 |
|------|------|
| 승인 | 다음 단계(분석) 진행 |
| 수정 필요 | 해당 요구 사항 수정 후 재확인 |
| 추가 발견 | 신규 요구 사항 등록 |
| 제외 결정 | 범위 외로 분류 |

## 4. 정보 요구사항 상세화 심화

### 4.1 분석 대상 자료 평가 기준

현행 시스템 관련 자료 수집 시 4가지 기준으로 평가:

| 기준 | 설명 |
|------|------|
| **유용성** | 문서를 분석 활동에 활용할 수 있는가 |
| **완전성** | 문서에 누락된 내용이 없는가 |
| **정확성** | 현재 운영 중인 시스템과 일치하는가 |
| **유효성** | 최신 문서인가 (구버전 아닌지 확인) |

### 4.2 프로세스 관점의 요구사항 상세화

**프로세스의 특징:**
- 실제 업무가 수행되는 행위로 시작·종료 시점이 명확
- 실행 횟수를 셀 수 있음
- 더 이상 분해할 수 없는 시점 = 기본 프로세스

**프로세스 도출 시 중요 원칙:**

| 원칙 | 설명 |
|------|------|
| 높은 응집도(Cohesion) | 하나의 프로세스는 단일 목적 수행 |
| 낮은 결합도(Coupling) | 프로세스 간 의존성 최소화 |

> 하위 프로세스 7개 초과 시 상위 프로세스 분리 고려

### 4.3 정보 요구사항 재검토 4가지 확인 기준

재검토(Review) 단계에서 산출물 누락·보완 사항을 도출하는 기준:

| 확인 기준 | 설명 |
|---------|------|
| **완전성** | 요구사항 누락 여부 확인 |
| **정확성** | 정확하게 표현되었는지 확인 |
| **일관성** | 표준화 준수 여부 확인 |
| **안정성** | 다른 업무 영역에 미치는 영향 여부 확인 |

### 4.4 재검토 절차

```
재검토 계획 수립 → 재검토 실시 → 보완 결과 확인
```

| 단계 | 주요 내용 |
|------|---------|
| 계획 수립 | 일자·장소·시간·참석대상·준비물 계획, 재검토 업무와 세부 시간 계획 |
| 재검토 실시 | 담당자별 업무 숙지, 배포된 산출물 **반드시 예습**, 해당 업무 관련자가 진행 |
| 보완 결과 확인 | 재검토 결과·보완목록 배포, 모두 반영되면 재검토 종료 |

## 출제 포인트

1. 분석 대상 선정 기준: 비즈니스 가치·우선순위·실현가능성·주제영역
2. 마스터 데이터: 기준 정보(고객·상품), 트랜잭션 데이터: 업무 처리 결과
3. 업무 규칙 4가지: 정의·사실·제약·파생 규칙
4. DFD 4개 구성 요소: 프로세스·데이터 저장소·외부 개체·데이터 흐름
5. 정보 요구 상세화: 데이터 항목 도출→관계 정의→업무 규칙→품질 요건
6. 인스펙션: 공식적, 독립 검토자, 결함 탐지 집중
7. 워크스루: 비공식적, 작성자 설명, 이해 공유
8. 확인 4대 목적: 정확성·완전성·일관성·실현가능성
9. 확인 결과 처리: 승인/수정 후 재확인/추가 등록/제외
10. 상세화 산출물: 상세 명세서·개념 데이터 모델·업무규칙 목록·DFD
