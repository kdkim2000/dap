# 4과목 2장: 개념 데이터 모델링

## 1. 개념 데이터 모델링 이해

**개념 데이터 모델링(Conceptual Data Modeling)** 은 업무에서 관리해야 할 핵심 데이터 개념과 이들 간의 관계를 비즈니스 관점에서 정의하는 단계이다. DBMS 독립적이며, 전사 관점의 데이터 구조를 파악하는 것이 목적이다.

| 특징 | 설명 |
|------|------|
| 관점 | 비즈니스·현업 관점 |
| 수준 | 높은 추상화 수준 |
| DBMS 독립성 | DBMS 기술에 무관 |
| 대상 독자 | 비즈니스 이해관계자·현업 담당자 |
| 산출물 | 개념 ERD, 주제 영역 정의서 |

---

## 2. 주제 영역 정의

**주제 영역(Subject Area)** 은 비즈니스 관점에서 유사한 데이터를 그룹화한 상위 분류 체계이다.

### 2.1 주제 영역의 특징

| 특징 | 설명 |
|------|------|
| 업무 기능과 유사 | 주요 업무 영역과 대응 |
| 3~10개 내외 | 너무 많거나 적지 않게 정의 |
| 배타적·완전 분류 | 엔터티는 하나의 주제 영역에만 속함 |

### 2.2 주제 영역 도출 방법

| 방법 | 내용 |
|------|------|
| 업무 기능 분석 | 비즈니스 기능을 기준으로 주제 영역 도출 |
| 정보 흐름 분석 | 데이터 흐름을 기준으로 관련 데이터 그룹화 |
| 조직 구조 분석 | 조직 단위·부서 기반으로 주제 영역 도출 |

**주제 영역 예시 (유통 기업):**

| 주제 영역 | 포함 엔터티 |
|---------|---------|
| 고객 | 고객, 고객 등급, 회원 |
| 상품 | 상품, 카테고리, 브랜드 |
| 주문 | 주문, 주문항목, 결제 |
| 재고 | 재고, 창고, 입출고 이력 |
| 마케팅 | 캠페인, 쿠폰, 이벤트 |

---

## 2a. 엔터티 분류 체계

블로그 교재에서는 엔터티를 3가지 관점으로 분류한다.

### 유형별 분류 (업무적 성격)

| 유형 | 설명 | 예시 |
|------|------|------|
| **유형(Tangible) 엔터티** | 물리적 형태가 있는 대상 | 고객, 직원, 상품 |
| **활동(Event) 엔터티** | 업무 행위나 사건으로 생성 | 주문, 계약, 접속 기록 |
| **개념(Conceptual) 엔터티** | 무형의 개념적 대상 | 계정과목, 보험상품, 성적 |

### 의존성에 따른 분류

| 유형 | 설명 |
|------|------|
| **독립(Fundamental) 엔터티** | 다른 엔터티에 의존하지 않고 독립적으로 존재 |
| **종속(Dependent) 엔터티** | 다른 엔터티의 존재에 의존하여 식별됨 |

### 발생 시점에 따른 분류

| 유형 | 설명 | 예시 |
|------|------|------|
| **키(Key) 엔터티** | 부모 엔터티 없음, 독립적으로 생성 | 상품, 고객 |
| **메인(Main) 엔터티** | 중심 업무를 수행하는 엔터티 | 주문 |
| **액션(Action) 엔터티** | 키·메인 엔터티의 행위 결과로 생성 | 주문 상세, 배송 |

### 관계 명칭 선정 기준

1. 현업에서 사용하는 간결한 **동사형** 사용
2. **현재 시제** 적용
3. 두 엔터티 간의 **업무적 연관성** 표현

### 속성 도출 방법

1. 현행 시스템 문서(화면, 보고서, 장표) 참고
2. 현업 장표 및 보고서 수집·분석
3. 현업 담당자 협의를 통한 도출

### 식별자 유형 (키의 종류)

| 유형 | 설명 |
|------|------|
| **후보키(Candidate Key)** | 유일성과 최소성을 모두 만족하는 키 |
| **기본키(Primary Key)** | 후보키 중 실제 튜플 식별에 사용되는 키 |
| **대체키(Alternate Key)** | 후보키 중 기본키로 선택되지 않은 키 |
| **외래키(Foreign Key)** | 다른 테이블의 기본키를 참조하는 키 |

## 3. 후보 엔터티 선정

**후보 엔터티(Candidate Entity)** 는 정보 요구 사항 분석에서 도출된 데이터 항목 중 독립적으로 존재할 수 있는 것들을 선별한 목록이다.

### 3.1 후보 엔터티 도출 방법

| 방법 | 내용 |
|------|------|
| 명사 분석 | 업무 설명에서 명사로 표현된 개념 추출 |
| 정보 요구 사항 분석 | 정보 요구 사항의 데이터 항목에서 도출 |
| 장표·화면 분석 | 업무 장표·화면에 반복 등장하는 개념 추출 |
| 인터뷰 결과 분석 | 이해관계자 인터뷰에서 언급된 주요 개념 |

### 3.2 엔터티 선정 기준

| 기준 | 설명 |
|------|------|
| 독립 존재성 | 다른 것에 의존하지 않고 독립적으로 존재 |
| 식별 가능성 | 고유하게 식별할 수 있는 속성 존재 |
| 2개 이상의 속성 | 단 하나의 속성만 있다면 엔터티가 아닐 가능성 |
| 업무 관련성 | 업무에서 실제로 관리·추적하는 대상 |

### 3.3 엔터티 후보 목록 관리

| 항목 | 내용 |
|------|------|
| 후보 엔터티명 | 한글·영문 명칭 |
| 설명 | 엔터티의 정의 |
| 출처 | 도출 근거 (어떤 요구 사항에서 나왔는가) |
| 포함 여부 | 채택/제외 결정 |

---

## 4. 핵심 엔터티 정의

후보 엔터티 중 업무에서 가장 중요한 **핵심 엔터티(Core Entity)** 를 선별하고 상세히 정의한다.

### 4.1 핵심 엔터티 선별 기준

| 기준 | 설명 |
|------|------|
| 비즈니스 핵심 | 주요 업무 기능에 필수적인 데이터 |
| 높은 활용도 | 여러 업무 영역에서 공통으로 사용 |
| 독립 존재 | 다른 엔터티에 종속되지 않음 |

### 4.2 엔터티 정의서

핵심 엔터티를 확정한 후 엔터티 정의서를 작성한다.

| 항목 | 내용 |
|------|------|
| 엔터티명 | 한글명, 영문명 |
| 정의 | 업무 관점의 의미 설명 |
| 식별자(후보) | 엔터티를 유일하게 구분하는 속성 |
| 주제 영역 | 속하는 주제 영역 |
| 관련 요구 사항 | 도출 근거 정보 요구 사항 ID |

## 5. 관계 정의

엔터티 간의 **관계(Relationship)** 를 정의하고 카디널리티·참여도를 결정한다.

### 5.1 관계 도출 방법

| 방법 | 내용 |
|------|------|
| 동사 분석 | 업무 설명에서 엔터티 간 연결 동사 추출 |
| 업무 규칙 분석 | 엔터티 간 관계를 정의하는 업무 규칙 |
| 정보 흐름 분석 | 데이터 흐름에서 연관 관계 파악 |

### 5.2 카디널리티(Cardinality)

| 관계 유형 | 설명 | 예시 |
|---------|------|------|
| 1:1 | 하나 대 하나 | 직원:사원증 |
| 1:M | 하나 대 다수 | 고객:주문 |
| M:N | 다수 대 다수 | 학생:강좌 |

**M:N 관계 처리:**
관계형 DB는 M:N을 직접 지원하지 않으므로, 교차 엔터티(연결 엔터티)를 추가하여 1:M × M:1로 분해한다.

```
학생 ──< 수강 >── 강좌
(1:M)        (M:1)
```

### 5.3 참여도(Participation)

| 구분 | 의미 | DB 구현 |
|------|------|---------|
| 필수(Mandatory) | 반드시 관계에 참여 | NOT NULL FK |
| 선택(Optional) | 관계에 참여 안 할 수도 있음 | NULL 허용 FK |

## 6. 개념 데이터 모델 작성

### 6.1 개념 ERD 작성 절차

```
1. 주제 영역 결정
2. 후보 엔터티 도출
3. 핵심 엔터티 확정
4. 관계 정의
5. 개념 ERD 작성
6. 검토 및 확인
```

### 6.2 개념 데이터 모델 검토 기준

| 검토 항목 | 확인 내용 |
|---------|---------|
| 엔터티 완전성 | 중요한 업무 데이터가 빠짐없이 정의되었는가 |
| 관계 정확성 | 엔터티 간 관계가 업무 규칙과 일치하는가 |
| 비즈니스 이해 용이성 | 현업 담당자가 이해할 수 있는 수준인가 |
| 주제 영역 일관성 | 모든 엔터티가 적절한 주제 영역에 속하는가 |

## 출제 포인트

1. 개념 데이터 모델링 목적: 핵심 데이터 개념·관계 정의, DBMS 독립적
2. 주제 영역: 유사 데이터의 상위 그룹화, 3~10개 내외
3. 주제 영역 도출: 업무 기능·정보 흐름·조직 구조 분석
4. 후보 엔터티 도출: 명사 분석·정보 요구·장표·인터뷰
5. 엔터티 선정 기준: 독립 존재성·식별 가능성·2개 이상 속성·업무 관련성
6. 핵심 엔터티: 비즈니스 핵심·높은 활용도·독립 존재
7. 관계 도출: 동사 분석·업무 규칙·정보 흐름 분석
8. M:N 관계 처리: 교차(연결) 엔터티 추가하여 1:M×M:1로 분해
9. 필수 참여 = NOT NULL FK, 선택 참여 = NULL 허용 FK
10. 개념 ERD 작성 순서: 주제영역→후보엔터티→핵심엔터티→관계→ERD→검토
