# 1과목 3장: 전사아키텍처 관리 및 활용

## 1. 전사아키텍처 관리 체계

EA 관리 체계는 수립된 아키텍처를 지속적으로 유지·통제하는 거버넌스 구조이다.

### 1.1 EA 거버넌스 구성 요소

| 구성 요소 | 설명 |
|---------|------|
| 의사결정 구조 | EA 관련 의사결정 조직과 권한 체계 |
| 프로세스 | 아키텍처 개발·검토·변경의 절차 |
| 정보 | EA 산출물과 메타데이터 관리 |
| 인력 | EA 담당 조직과 역할 |

### 1.2 EA 관리 조직

| 역할 | 책임 |
|------|------|
| 수석 아키텍트(Chief Architect) | EA 총괄 책임, 전략 방향 결정 |
| EA 위원회(EA Board) | 아키텍처 방향·원칙 결정 |
| 아키텍처 검토 위원회(ARB) | 프로젝트 아키텍처 준수 여부 검토 |
| 도메인 아키텍트 | BA/DA/AA/TA 도메인별 담당 |
| 프로젝트 아키텍트 | 개별 프로젝트 아키텍처 설계·관리 |

### 1.3 아키텍처 거버넌스 프로세스

| 프로세스 | 설명 |
|---------|------|
| 아키텍처 개발 | EA 수립·개정 절차 |
| 아키텍처 검토 | 프로젝트 적합성 검토(ARB) |
| 변경 관리 | 아키텍처 변경 요청 통제 |
| 준수 관리 | 표준·원칙 준수 여부 모니터링 |

### 1.4 아키텍처 준수 관리

| 단계 | 내용 |
|------|------|
| 기준 수립 | 아키텍처 원칙·표준·지침 정의 |
| 프로젝트 검토 | ARB의 아키텍처 준수 검토 |
| 편차 관리 | 불가피한 편차 공식 승인 |
| 사후 감사 | 구현 결과의 EA 준수 여부 확인 |

**아키텍처 편차(Deviation):**

| 유형 | 설명 |
|------|------|
| 임시 편차(Temporary) | 특정 기간 동안만 허용 |
| 영구 편차(Permanent) | 지속적으로 허용 |

## 2. 전사아키텍처 관리 시스템

EA 관리 시스템은 아키텍처 정보를 체계적으로 저장·관리·제공하는 도구이다.

### 2.1 EA 리포지터리(Repository)

아키텍처 산출물을 중앙에서 저장·관리하는 시스템.

| 저장 대상 | 예시 |
|---------|------|
| 아키텍처 모델 | ERD, 아키텍처 다이어그램 |
| 표준 문서 | 기술 표준, 데이터 표준, 코드 표준 |
| 원칙 | 아키텍처 원칙 문서 |
| 이행 산출물 | 아키텍처 준수 보고서 |

**TOGAF 리포지터리 4대 구성 요소:**
- 아키텍처 메타모델(Metamodel)
- 아키텍처 역량(Capability)
- 아키텍처 랜드스케이프(Landscape)
- 표준 정보 기반(Standards Information Base)

### 2.2 EA 포털(Portal)

조직 구성원이 EA 정보에 접근하는 웹 기반 인터페이스.

| 기능 | 설명 |
|------|------|
| 산출물 검색 | 아키텍처 문서·모델 검색 |
| 표준 조회 | 기술·데이터 표준 확인 |
| 변경 요청 | 아키텍처 변경 요청 접수 |
| 현황 조회 | AS-IS/TO-BE 아키텍처 현황 |

> **핵심**: EA 리포지터리는 산출물 저장·관리 시스템, EA 포털은 구성원의 정보 접근 인터페이스로 역할이 다르다.

### 2.3 EA 관리 시스템 핵심 기능

| 기능 | 설명 |
|------|------|
| 형상 관리 | 아키텍처 변경 이력 추적 |
| 관계 분석 | 구성 요소 간 의존 관계 파악 |
| 영향 분석 | 변경 시 영향 범위 자동 분석 |
| 보고서 생성 | 현황 분석 보고서 자동 생성 |

### 2.4 EA 성과 관리 지표

| 지표 유형 | 예시 |
|---------|------|
| EA 수립 지표 | 아키텍처 완성도(%), 문서 최신화율 |
| EA 활용 지표 | 포털 방문 수, 검토 요청 건수 |
| EA 준수 지표 | 표준 준수율, 미준수 프로젝트 수 |
| IT 투자 효과 | 중복 시스템 감소, 비용 절감액 |

## 3. 전사아키텍처 활용

EA 활용은 수립된 아키텍처를 실제 IT 의사결정과 업무에 적극적으로 사용하는 것이다. 성숙도 6단계(활용, Leveraged)가 최고 수준이다.

### 3.1 IT 투자 의사결정 지원

| 활용 방법 | 내용 |
|---------|------|
| 신규 투자 타당성 검토 | EA 기반으로 신규 시스템 필요성·중복성 판단 |
| 아키텍처 적합성 평가 | 프로젝트가 EA 방향과 부합하는지 평가 |
| 기술 표준 적용 | 표준화된 기술 컴포넌트 우선 사용 유도 |

### 3.2 프로젝트 관리 지원

- **범위 정의**: EA 기반으로 프로젝트 영향 범위 파악
- **설계 기준**: 아키텍처 원칙·표준을 설계 기준으로 적용
- **ARB 검토**: 프로젝트 아키텍처의 EA 준수 여부 검토

### 3.3 정보화 전략 계획(ISP) 연계

| EA 산출물 | ISP 활용 |
|---------|---------|
| AS-IS 아키텍처 | 현황 분석 기반 자료 |
| TO-BE 아키텍처 | 미래 정보화 방향 |
| 이행 계획 | ISP 로드맵 수립 기준 |

### 3.4 EA 활용 수준과 성숙도

| 성숙도 단계 | 활용 수준 |
|--------|---------|
| 4단계(완료) | EA 수립은 완료, 활용 미흡 |
| 5단계(유지보수) | EA 정보를 최신화하여 참조 수준 |
| 6단계(활용) | EA를 의사결정에 적극 활용 (최고 수준) |

### 3.5 EA 지식 관리

| 방법 | 내용 |
|------|------|
| 아키텍처 패턴 라이브러리 | 검증된 설계 패턴 축적·공유 |
| 사례(Case Study) 관리 | 성공/실패 사례 기록 |
| 역량 개발 교육 | 아키텍트 역량 향상 프로그램 |

## 4. 데이터 거버넌스

### 4.1 데이터 거버넌스 등장 배경

데이터량의 급증과 함께 "항상 이용 가능한 데이터의 시대"에 접어들면서, 데이터가 기업의 귀중한 자산으로 인식되었다. 높은 품질의 데이터를 유지·관리하기 위한 체계의 필요성으로 등장했다.

### 4.2 데이터 거버넌스 정의와 핵심 관점

**정의**: "기업이나 조직의 목적에 맞게 높은 품질의 데이터를 사용할 수 있도록 정의한 체계"

| 핵심 관점 | 설명 |
|---------|------|
| 책임(Accountability) | 조직의 모든 계층에서 데이터 책임 정의 |
| 일관성(Consistency) | 표준화된 절차를 통한 데이터 일관성 유지 |
| 적응성(Adaptability) | 조직의 전략적 방향에 맞는 적응 |

### 4.3 데이터 거버넌스 필수 요소

1. 데이터 요소에 대한 표준화된 명칭 및 생성 규칙
2. DB 시스템에서의 업무 데이터 물리적 위치 정의
3. 조직에 맞는 데이터 품질 규칙
4. 데이터 사용 규칙
5. 데이터 통제자·관리자·소유자의 적절한 통제

> **핵심**: 데이터 거버넌스는 IT 관련 조직만이 아닌 **전사적인 노력**이 필요하다.

### 4.4 데이터 거버넌스 역할 정의

| 역할 | 설명 |
|------|------|
| **데이터 통제자(Data Controller)** | 전사적 데이터 거버넌스 운영 총괄 |
| **데이터 관리자(Data Administrator)** | 데이터 품질 및 무결성 책임 |
| **데이터 소유자(Data Owner)** | 특정 데이터 도메인의 접근 권한 및 품질 책임 |

### 4.5 데이터 거버넌스 주요 고려사항

| 고려사항 | 내용 |
|---------|------|
| 보안성 | 역할별 접근 권한 부여 및 민감 데이터 보호 |
| 무결성 | 데이터 처리 원칙 정의 및 모니터링 |
| 데이터 품질 | 최신 상태를 정확히 반영하는 관리 체계 |
| 오너십(Ownership) | 소유권 부여로 책임 확보 |

### 4.6 데이터 아키텍트(DA)의 역량

| 역량 유형 | 설명 |
|---------|------|
| **리더십 역량** | 전사적 공감대 형성 및 변화 관리 |
| **기술적 역량** | 이론과 실제 구축 능력 (모델링·표준화·DB 지식) |
| **활용 역량** | 데이터 품질 유지로 업무 효과성 향상 |

### 4.7 현대적 거버넌스 체계 수립 고려사항

1. **전사 범위 정의**: 내부 및 외부 데이터 원천 포함
2. **민감 데이터 처리**: 개인정보 외 유출 방지 대상 식별
3. **새로운 품질 관리**: 빅데이터에 적합한 검증 방식 적용
4. **카탈로그/메타데이터 관리**: 데이터 위치 및 정보 체계화
5. **적재 및 용량 관리**: 지속적 점검 및 예측

## 출제 포인트

1. ARB: 프로젝트 아키텍처 준수 여부 검토 기구
2. EA 거버넌스 4대 구성 요소: 의사결정 구조·프로세스·정보·인력
3. EA 리포지터리: 산출물 중앙 저장·관리 / EA 포털: 구성원 접근 인터페이스
4. TOGAF 리포지터리 구성: 메타모델·역량·랜드스케이프·표준 정보 기반
5. 아키텍처 편차: 표준 불준수 시 공식 승인 예외 (임시 편차 / 영구 편차)
6. EA 활용 지표: 포털 방문 수, 검토 요청 건수
7. EA 준수 지표: 표준 준수율, 미준수 프로젝트 수
8. 성숙도 6단계(Leveraged): EA를 의사결정에 적극 사용 = 최고 수준
9. EA와 ISP 연계: AS-IS→현황, TO-BE→미래방향, 이행계획→로드맵
10. EA 관리 시스템 기능: 형상관리·관계분석·영향분석·보고서생성
11. 수석 아키텍트(Chief Architect): EA 총괄 책임자
12. 아키텍처 패턴 라이브러리: 검증된 설계 패턴 재사용 촉진
