# 1과목 2장: 전사아키텍처 구축

## 1. 전사아키텍처 방향 수립

EA 방향 수립은 조직의 비즈니스 전략에서 EA 비전·원칙·목표를 정의하는 단계이다. EA 구축 전 반드시 선행되어야 한다.

### 1.1 EA 비전 및 목표 수립

| 항목 | 내용 |
|------|------|
| EA 비전 | EA를 통해 달성하고자 하는 미래 모습 |
| EA 목표 | 비전을 구체화한 측정 가능한 목표 |
| EA 범위 | 아키텍처 적용 대상 조직·시스템 범위 |
| EA 전략 | 목표 달성을 위한 접근 방법 |

### 1.2 EA 원칙 정의

EA 원칙은 아키텍처 수립·운영 시 준수해야 하는 고수준 지침이다.

**EA 원칙 구성 요소:**

| 구성 | 설명 |
|------|------|
| 원칙명 | 간결한 명칭 |
| 설명 | 원칙의 의미 |
| 근거 | 원칙 채택 이유 |
| 영향 | 원칙 적용 시 의미 |

**도메인별 원칙 예시:**
- 비즈니스: "업무는 IT가 아닌 비즈니스 요구를 중심으로 설계한다"
- 데이터: "데이터는 전사 자산이다"
- 애플리케이션: "개방형 표준을 사용한다"
- 기술: "표준 기술 컴포넌트를 우선 활용한다"

### 1.3 접근 방식 결정

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| Top-Down | 전략 → 아키텍처 순 | 비즈니스 정렬성 높음 | 시간·비용 소요 |
| Bottom-Up | 현행 시스템 분석 → 아키텍처 도출 | 현실 반영도 높음 | 비즈니스 정렬성 부족 |
| Hybrid | Top-Down + Bottom-Up 결합 | 균형적 접근 | 조정 복잡 |

## 2. 전사아키텍처 정보 구성 정의

EA 정보 구성 정의는 어떤 아키텍처 정보를 어떻게 구조화할지를 결정하는 단계이다.

### 2.1 아키텍처 뷰와 뷰포인트

| 개념 | 설명 |
|------|------|
| 아키텍처 뷰(View) | 특정 관심사에 따른 시스템의 표현 |
| 뷰포인트(Viewpoint) | 특정 역할(이해관계자)이 관심 갖는 관점 |
| 아키텍처 산출물(Artifact) | 뷰를 구체화한 문서·모델 |

### 2.2 EA 산출물 체계 정의

| 도메인 | 주요 산출물 |
|--------|---------|
| BA | 업무 기능 분류체계, 업무 흐름도, 조직도 |
| DA | 개념/논리/물리 데이터 모델, 데이터 표준, 데이터 흐름도 |
| AA | 응용 기능 분류체계, 인터페이스 목록, 응용 구성도 |
| TA | 기술 참조 모델, 시스템 구성도, 네트워크 구성도 |

### 2.3 아키텍처 메타모델(Metamodel)

아키텍처 요소와 이들 간의 관계를 정의한 모델로, EA 리포지터리의 기반이 된다.

```
비즈니스 프로세스 ──지원── 애플리케이션 ──사용── 데이터
        │                      │
      수행                    실행
        │                      │
      조직                   기술 플랫폼
```

### 2.4 EA 정보 표현 원칙

| 원칙 | 내용 |
|------|------|
| 일관성 | 동일 개념을 동일 방식으로 표현 |
| 완전성 | 필요한 모든 정보 포함 |
| 정확성 | 현실을 올바르게 반영 |
| 추적성 | 요소 간 연결 관계 명확화 |

## 3. 전사아키텍처 정보 구축

EA 정보 구축은 정의된 구성 체계에 따라 실제 아키텍처 정보를 수집·분석·문서화하는 단계이다.

### 3.1 현행(AS-IS) 아키텍처 정보 수집

| 수집 방법 | 내용 |
|---------|------|
| 인터뷰 | 현업·IT 담당자 면담으로 현황 파악 |
| 문서 검토 | 기존 설계서·운영 매뉴얼 분석 |
| 설문 | 다수 이해관계자 대상 현황 조사 |
| 자동화 도구 | EA 관리 도구로 시스템 정보 자동 수집 |

### 3.2 현행 아키텍처 분석 항목

| 분석 항목 | 내용 |
|---------|------|
| 중복 시스템 식별 | 동일 기능을 수행하는 중복 시스템 파악 |
| 인터페이스 현황 | 시스템 간 데이터 교환 현황 |
| 기술 표준 현황 | 현재 사용 중인 기술 표준 |
| 데이터 현황 | 주요 데이터 항목과 흐름 |

### 3.3 목표(TO-BE) 아키텍처 수립

| 구분 | 내용 |
|------|------|
| 목표 아키텍처 설계 | 비즈니스 전략과 AS-IS 갭 기반으로 목표 상태 설계 |
| 이행 계획 수립 | 단계별(Phase) 전환 계획과 우선순위 결정 |
| 아키텍처 검토 | ARB 검토·승인 |

### 3.4 갭(Gap) 분석

| 절차 | 내용 |
|------|------|
| AS-IS 파악 | 현행 아키텍처 요소 목록화 |
| TO-BE 정의 | 목표 아키텍처 요소 정의 |
| 갭 식별 | 차이 분석: 추가·변경·제거 요소 |
| 이행 방안 | 갭 해소를 위한 프로젝트·조치 도출 |

### 3.5 EA 구축 산출물 검증

| 검증 항목 | 방법 |
|---------|------|
| 완전성 | 모든 도메인 산출물 존재 여부 확인 |
| 일관성 | 도메인 간 산출물 연계 관계 확인 |
| 정확성 | 현업 검토를 통한 내용 정확성 확인 |
| 추적성 | 비즈니스 요구 → 기술까지 연결 가능 여부 |

## 4. 데이터 아키텍처 방향 수립 심화

### 4.1 데이터 아키텍처 환경분석

EA 구축 전 내·외부 환경을 분석하여 구축 방향의 근거를 마련한다.

| 분석 요인 | 주요 내용 |
|---------|---------|
| 외부 분석요인 | 기술 이슈, 트렌드, 법제도 변화 등 외부 동향 |
| 내부 분석요인 | 외부 환경에 대응하는 내부 시스템·조직 동향 |
| 이해관계자 분석요인 | 데이터 관련 현업의 불만, 요구사항 |

### 4.2 아키텍처 매트릭스(Architecture Matrix)

2차원 매트릭스로 의사결정 유형과 정보 유형을 구성하며, 각 셀에서 필요한 산출물을 정의한다.

| 구성 축 | 세부 유형 |
|--------|---------|
| 의사결정 유형 (행) | 계획자, 책임자(소유자), 설계자, 개발자 |
| 정보 유형 (열) | 업무, 데이터, 응용, 기술 |

**아키텍처 매트릭스 정의 시 고려사항:**
- 모든 이해관계자의 동의 필수
- 선진 모범 사례 참조 및 조직에 맞게 조정
- 아키텍처 원칙 및 표준 준수성 강화
- 타 도메인(BA/DA/AA/TA)과의 상호 연계성 고려

### 4.3 현행 데이터 분석 4가지 방법

현행 데이터아키텍처 정보를 구축할 때 수행하는 분석:

| 분석 방법 | 주요 내용 |
|---------|---------|
| 1. 데이터 현황 분석 | DB 개체 및 스토리지 사용 현황 파악, 완전성·우선순위·정합성·유효성 분석 |
| 2. 데이터 표준화 현황 분석 | 명칭·도메인·코드 사용의 일관성 파악 |
| 3. 문서화 현황 및 정합성 분석 | 설계 문서와 실제 DB 개체의 일치성 확인 |
| 4. 설계 및 관리 현황 분석 | 현행화 상태 확인 및 데이터 관리 체계 파악 |

## 출제 포인트

1. EA 방향 수립: 비전·목표·범위·전략 정의
2. EA 원칙 구성 요소: 원칙명·설명·근거·영향
3. "데이터는 전사 자산이다" → DA 원칙 예시
4. 아키텍처 뷰(View): 특정 관심사에 따른 시스템 표현
5. 아키텍처 메타모델: EA 요소와 관계를 정의한 모델, 리포지터리 기반
6. EA 정보 구축 수집 방법: 인터뷰·문서검토·설문·자동화도구
7. AS-IS 분석 항목: 중복시스템·인터페이스·기술표준·데이터 현황
8. 갭 분석: AS-IS와 TO-BE의 차이 → 추가·변경·제거 요소 도출
9. Top-Down: 전략→아키텍처, Bottom-Up: 현행 분석→아키텍처
10. Hybrid 방식: 실무에서 가장 많이 사용
11. EA 산출물 검증: 완전성·일관성·정확성·추적성
12. ARB: 구축 산출물 검토·승인 기구
