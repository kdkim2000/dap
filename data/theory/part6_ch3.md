# 6과목 3장: 데이터 관리 프로세스 이해

## 1. 데이터 관리 정책

**데이터 관리 정책**은 조직의 데이터를 체계적으로 관리하기 위한 방침과 원칙이다.

### 1.1 데이터 관리 정책 구성 요소

| 구성 요소 | 설명 |
|------|------|
| 데이터 관리 원칙 | 데이터 관리의 기본 방향과 기준 |
| 데이터 관리 조직 | 역할·책임 체계 (오너, 스튜어드, DBA) |
| 데이터 관리 절차 | 표준 수립, 변경 관리, 품질 검증 절차 |
| 데이터 관리 도구 | 메타데이터 관리 시스템, 품질 측정 도구 |

### 1.2 데이터 거버넌스(Data Governance)

데이터 관리에 관한 의사결정 권한·책임·프로세스의 체계.

| 구성 요소 | 설명 |
|------|------|
| 데이터 위원회 | 데이터 관련 최고 의사결정 기구 |
| 데이터 표준화 위원회 | 표준 수립·변경 승인 |
| 데이터 오너 | 업무 데이터 책임 보유 |
| 데이터 스튜어드 | 일상적 데이터 품질 관리 |

---

## 2. 데이터 표준 관리

**데이터 표준 관리**는 조직의 데이터 표준(단어·용어·코드·도메인)을 정의·유지·변경하는 프로세스이다.

### 2.1 데이터 표준 관리 절차

```
표준 요청 → 표준 검토 → 표준 확정 → 표준 공표 → 표준 적용 → 표준 유지보수
```

### 2.2 표준 변경 관리

| 단계 | 내용 |
|------|------|
| 변경 요청 | 표준 변경 필요성 제기 |
| 영향 분석 | 변경 시 영향받는 시스템·모델 분석 |
| 변경 승인 | 데이터 표준화 위원회 승인 |
| 변경 적용 | 관련 시스템·모델에 변경 적용 |
| 변경 검증 | 적용 결과 확인 |

---

## 3. 요구사항 관리

**데이터 요구사항 관리**는 사용자의 데이터 관련 요구사항을 수집·분석·추적·관리하는 프로세스이다.

### 3.1 요구사항 관리 절차

| 단계 | 활동 |
|------|------|
| 요구사항 수집 | 인터뷰, 설문, 문서 검토 |
| 요구사항 정리 | 표준 형식으로 문서화 |
| 요구사항 분석 | 실현 가능성·모순·중복 검토 |
| 요구사항 확인 | 이해관계자 검토·승인 |
| 추적성 관리 | 요구사항↔데이터 모델 연결 유지 |

### 3.2 요구사항 추적성(Traceability)

- 요구사항이 데이터 모델의 어느 부분으로 구현되었는지 추적 가능해야 함
- 요구사항 변경 시 영향받는 모델 범위를 식별

---

## 4. 데이터 모델 관리

**데이터 모델 관리**는 개념·논리·물리 데이터 모델의 변경·이력·품질을 관리하는 프로세스이다.

### 4.1 데이터 모델 변경 관리

| 변경 유형 | 예시 | 처리 방법 |
|------|------|------|
| 엔터티 추가 | 새 업무 엔터티 발생 | 모델 추가 후 영향 분석 |
| 속성 변경 | 컬럼 길이 변경 | DDL 변경 및 데이터 마이그레이션 |
| 관계 변경 | 1:N → M:N | 중간 테이블 추가 |
| 엔터티 삭제 | 사용하지 않는 테이블 | 영향 분석 후 단계적 삭제 |

### 4.2 데이터 모델 형상 관리(Configuration Management)

- 모델 버전 이력 관리
- 변경 전·후 모델 비교 가능
- 이전 버전으로 롤백 가능

---

## 5. 데이터 흐름 관리

**데이터 흐름 관리**는 시스템 간 데이터 이동·변환·연계를 관리하는 프로세스이다.

### 5.1 데이터 흐름도(Data Flow Diagram)

시스템 간 데이터 흐름을 시각적으로 표현한 다이어그램.

| 구성 요소 | 설명 |
|------|------|
| 프로세스 | 데이터를 처리하는 기능 |
| 데이터 저장소 | 데이터가 저장되는 곳 |
| 외부 엔터티 | 시스템 외부의 데이터 소스·목적지 |
| 데이터 흐름 | 데이터의 이동 경로 |

### 5.2 인터페이스 관리

| 유형 | 설명 |
|------|------|
| 실시간 인터페이스 | 즉시 데이터 전송 (API, MQ) |
| 배치 인터페이스 | 일괄 데이터 전송 (FTP, ETL) |
| DB 연계 | 데이터베이스 링크를 통한 직접 연계 |

### 5.3 데이터 계보 (Data Lineage)

데이터가 **어디서 생성되어, 어떤 변환 과정을 거쳐, 어디로 이동**했는지 추적하는 관리 체계.

```
[소스 시스템 A] → [ETL 변환] → [데이터 웨어하우스] → [데이터 마트] → [리포트]
     ↑ 계보(Lineage) 추적: 어디서 왔는지, 어떻게 변환됐는지 전체 경로 기록
```

| 계보 유형 | 설명 |
|------|------|
| **순방향 계보** | 소스에서 시작 → 데이터가 어디로 흘러가는지 추적 |
| **역방향 계보** | 목적지에서 시작 → 데이터가 어디서 왔는지 추적 |

**Data Lineage의 필요성**:
- 데이터 품질 문제 발생 시 원인 데이터 소급 추적
- 규제 준수(감사) — 특정 데이터가 어떤 경로로 처리됐는지 증명
- 시스템 변경 시 영향 범위 파악 (역방향 계보 활용)

### 5.4 CDC (Change Data Capture)

소스 데이터베이스에서 변경(INSERT/UPDATE/DELETE)된 데이터만 **실시간으로 감지**하여 목적지 시스템에 전달하는 기법.

| 방식 | 설명 | 특징 |
|------|------|------|
| **로그 기반 CDC** | DBMS의 트랜잭션 로그(Redo Log) 분석 | 성능 영향 최소, 가장 일반적 |
| **트리거 기반 CDC** | DML 트리거로 변경 감지 | 구현 쉬움, 성능 부하 가능 |
| **타임스탬프 기반** | 변경 타임스탬프 컬럼 비교 | 간단하지만 삭제 감지 어려움 |
| **스냅샷 비교** | 전체 데이터 주기적 비교 | 구현 쉬움, 대용량 비효율 |

```
[소스 DB] → [CDC 엔진] → [메시지 큐(Kafka)] → [목적지(DW/마트)]
              ↑ 트랜잭션 로그에서 변경분만 포착
```

**CDC 활용 사례**:
- 운영 DB → DW 실시간 동기화
- 마이크로서비스 간 이벤트 기반 데이터 동기화
- 배치를 실시간으로 전환 (준실시간 DW 구축)

## 6. 데이터베이스 관리

**데이터베이스 관리**는 데이터베이스의 운영·성능·보안을 관리하는 프로세스이다.

### 6.1 DBA(Database Administrator) 주요 역할

| 역할 | 내용 |
|------|------|
| 스키마 관리 | 테이블·인덱스·뷰 생성·변경·삭제 |
| 성능 관리 | 쿼리 튜닝, 인덱스 최적화 |
| 백업·복구 | 정기 백업, 장애 시 복구 |
| 보안 관리 | 접근 권한, 감사 로그 |
| 용량 관리 | 스토리지 모니터링, 파티셔닝 |

### 6.2 RTO와 RPO (복구 목표 지표)

| 지표 | 전체 이름 | 의미 |
|------|------|------|
| **RTO** | Recovery Time Objective | 장애 발생 후 서비스가 복구되기까지 **허용되는 최대 시간** |
| **RPO** | Recovery Point Objective | 장애 시 복구 가능한 **가장 최근 시점** — 허용되는 데이터 손실 범위 |

```
장애 발생
    │
    │← RPO ──────────────┤    복구 목표 시점 (예: 1시간 전 백업)
    │
    │                         데이터 손실 허용 범위
    │
    │← RTO →│                 서비스 복구 완료까지 최대 허용 시간
```

| 서비스 유형 | RTO | RPO | 대응 방안 |
|------|:---:|:---:|------|
| 금융 핵심 시스템 | < 1분 | 0 (무손실) | 실시간 복제, 2-Phase Commit |
| 전자상거래 | < 1시간 | < 15분 | Hot Standby, 로그 백업 |
| 내부 정보 시스템 | < 24시간 | < 4시간 | 일일 전체 백업 |

> **핵심**: RTO와 RPO가 짧을수록 고가용성 요구사항이 높아져 비용 증가. 업무 중요도에 따라 적절히 설정.

## 7. 데이터 활용 관리

**데이터 활용 관리**는 데이터를 올바르게 분석·리포팅·의사결정에 활용하기 위한 관리 프로세스이다.

### 7.1 데이터 활용 유형

| 유형 | 설명 |
|------|------|
| OLTP(Online Transaction Processing) | 실시간 업무 처리 시스템 |
| OLAP(Online Analytical Processing) | 다차원 분석, 의사결정 지원 |
| 데이터 웨어하우스(DW) | 대용량 분석용 통합 데이터 저장소 |
| 데이터 마트(Data Mart) | DW에서 특정 주제 영역 분리 |

### 7.2 ETL(Extract, Transform, Load)

운영 DB → DW로 데이터를 이동하는 프로세스.
- **Extract**: 소스 데이터 추출
- **Transform**: 데이터 변환·정제·통합
- **Load**: 목적지(DW)에 적재

## 출제 포인트

1. **데이터 거버넌스 4요소**: 위원회·표준화위원회·오너·스튜어드
2. **표준 변경 프로세스**: 요청→검토→영향분석→승인→적용→검증
3. **데이터 흐름도 4요소**: 프로세스·데이터저장소·외부엔터티·데이터흐름
4. **OLTP vs OLAP**: 실시간 업무 처리 vs 다차원 분석
5. **ETL**: Extract(추출)→Transform(변환)→Load(적재)
6. **데이터 마트**: DW에서 특정 주제 영역 분리
7. **데이터 계보(Data Lineage)**: 데이터가 어디서 생성→어떤 변환→어디로 이동했는지 추적
8. **순방향 vs 역방향 계보**: 소스→목적지 추적 vs 목적지→소스 소급 추적
9. **CDC(Change Data Capture)**: 변경된 데이터만 실시간 감지·전달 — 로그 기반이 가장 일반적
10. **CDC 활용**: 운영DB→DW 실시간 동기화, 준실시간 DW 구축
11. **RTO(Recovery Time Objective)**: 장애 후 서비스 복구까지 허용되는 최대 시간
12. **RPO(Recovery Point Objective)**: 허용되는 데이터 손실 범위(복구 가능 최근 시점)
13. **RTO·RPO 관계**: 짧을수록 고가용성 필요, 비용 증가
14. **DBA 주요 역할**: 스키마·성능·백업·보안·용량 관리
15. **요구사항 추적성**: 요구사항↔데이터 모델 연결
16. **형상 관리**: 모델 버전 이력 관리, 롤백 가능
