# 4과목 3장: 논리 데이터 모델링

## 0. 논리 데이터 모델링의 기능 4가지

논리 데이터 모델링이 조직에 제공하는 핵심 기능:

1. **전사적 통합 데이터 체계 구축**: 조직 전체의 일관된 데이터 구조 정립
2. **데이터 일관성 및 정확성 규칙 도출**: 업무 규칙과 무결성 조건 명세화
3. **사용자와의 명확한 의사소통 수단**: 비즈니스 언어로 표현된 논리 ERD로 현업과 소통
4. **안정적인 데이터베이스 설계 토대**: 물리 모델링의 기반이 되는 논리 구조 수립

---

## 1. 논리 데이터 모델링 이해

**논리 데이터 모델링(Logical Data Modeling)**은 개념 데이터 모델을 기반으로 속성, 엔터티 관계, 이력 관리 등을 상세히 정의하는 단계이다. DBMS에 독립적이며, 데이터베이스 설계를 위한 논리적 구조를 결정한다.

| 특징 | 설명 |
|------|------|
| 관점 | 데이터 관리 관점 (중간 수준 추상화) |
| DBMS 독립성 | DBMS 기술에 무관 |
| 표현 | IE 또는 Barker 표기법의 ERD |
| 대상 독자 | DA, 시스템 설계자 |
| 산출물 | 논리 ERD, 엔터티 정의서, 속성 정의서 |

**개념 → 논리 모델링 전환:**

| 개념 모델 | 논리 모델 |
|---------|---------|
| 엔터티(개략적) | 엔터티(상세, 정규화된) |
| 관계(선택적 기술) | 관계(카디널리티·참여도 상세화) |
| 핵심 속성 일부 | 모든 속성 완전 정의 |
| 식별자(후보) | 기본 키, 외래 키 확정 |

---

## 2. 속성 정의

**속성(Attribute)** 은 엔터티의 특성을 나타내는 데이터 항목이다. 논리 모델링에서 모든 속성을 빠짐없이 정의해야 한다.

### 2.1 속성 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| 기본 속성 | 업무에서 자연스럽게 도출된 속성 | 고객명, 전화번호 |
| 설계 속성 | 모델링·설계 과정에서 필요로 추가된 속성 | 코드 컬럼 |
| 파생 속성 | 다른 속성 값으로 계산·유도 | 나이(생년월일로 계산), 총액 |

### 2.2 속성 정의서 작성

| 항목 | 내용 |
|------|------|
| 속성명 | 한글명, 영문명 |
| 속성 설명 | 업무 의미 |
| 도메인 | 데이터 타입, 길이, 형식 |
| NULL 여부 | 필수(NOT NULL) / 선택(NULL) |
| 기본값 | Default 값 |
| 업무 규칙 | 허용 값, 유효성 규칙 |

### 2.3 도메인(Domain) 정의

속성이 가질 수 있는 값의 범위·타입·형식을 정의한 표준.

| 도메인 | 데이터 타입 | 길이 | 형식 |
|--------|---------|------|------|
| 날짜 | DATE | — | YYYY-MM-DD |
| 금액 | NUMBER | 15,2 | 소수점 2자리 |
| 코드 | VARCHAR | 10 | 영숫자 |
| 이름 | VARCHAR | 100 | 한글 허용 |

### 2.4 복합 속성(Composite Attribute) 처리

여러 값을 포함하는 복합 속성은 개별 원자 속성으로 분해해야 한다 (1NF 조건).

```
주소 → 우편번호 + 시도 + 시군구 + 상세주소
```

## 3. 엔터티 상세화

개념 모델의 엔터티를 논리 모델에서 완전하게 상세화하는 단계이다.

### 3.1 슈퍼타입/서브타입 모델링

공통 속성을 상위 엔터티(슈퍼타입)에, 특수 속성을 하위 엔터티(서브타입)에 정의하는 방식.

```
직원(슈퍼타입)
  ├── 정규직(서브타입)
  ├── 계약직(서브타입)
  └── 파견직(서브타입)
```

| 개념 | 설명 |
|------|------|
| 슈퍼타입(Super Type) | 공통 속성을 가진 상위 엔터티 |
| 서브타입(Sub Type) | 특수 속성을 가진 하위 엔터티 |
| 배타적(Exclusive) | 하나의 서브타입에만 속함 |
| 포함적(Inclusive) | 여러 서브타입에 동시 속함 가능 |

### 3.2 관계 상세화

개념 모델의 관계를 논리 모델에서 외래 키(FK)로 구현:

| 관계 유형 | 구현 방법 |
|---------|---------|
| 1:1 | FK를 어느 한쪽 테이블에 추가 |
| 1:M | 다(Many) 쪽에 FK 추가 |
| M:N | 연결 테이블(교차 엔터티) 추가 |

### 3.3 식별 관계 vs 비식별 관계

| 구분 | 설명 | 특징 |
|------|------|------|
| 식별 관계 | 부모 PK → 자식 PK 일부 구성 | 자식이 부모 없이 존재 불가 |
| 비식별 관계 | 부모 PK → 자식 일반 속성(FK) | 자식이 독립적 존재 가능 |

## 3a. 참조 무결성 규칙 정의

부모-자식 관계 엔터티의 입력 및 삭제 시 적용할 규칙을 논리 모델링 단계에서 정의한다.

### 입력 규칙 (자식 엔터티 데이터 입력 시)

| 규칙 | 설명 |
|------|------|
| **Dependent** | 부모 엔터티 인스턴스가 존재할 때만 자식 입력 허용 |
| **Automatic** | 부모 인스턴스가 없으면 자동으로 부모를 생성 |
| **Nullify** | 부모 인스턴스가 없으면 참조키를 NULL로 처리 |
| **Default** | 부모 인스턴스가 없으면 참조키를 기본값으로 처리 |
| **Customized** | 특정 검증 조건을 만족할 때만 입력 허용 |
| **No Effect** | 조건 없이 항상 입력 허용 |

### 삭제 규칙 (부모 엔터티 데이터 삭제 시)

| 규칙 | 설명 |
|------|------|
| **Restrict** | 자식 엔터티 인스턴스가 없을 때만 부모 삭제 허용 |
| **Cascade** | 부모 삭제 시 대응되는 자식도 자동으로 삭제 |
| **Nullify** | 부모 삭제 시 자식의 참조키를 NULL로 처리 |
| **Default** | 부모 삭제 시 자식의 참조키를 기본값으로 처리 |
| **Customized** | 특정 검증 조건을 만족할 때만 삭제 허용 |
| **No Effect** | 조건 없이 항상 삭제 허용 |

## 4. 이력관리 정의

**이력 관리(History Management)** 는 데이터의 변경 내용을 시간 순으로 기록하여 과거 상태를 조회할 수 있게 하는 것이다.

### 4.0 이력 관리의 3가지 종류

| 종류 | 설명 | 예시 |
|------|------|------|
| **발생 이력 데이터** | 데이터 발생 시마다 이력 기록 | 웹사이트 접속 기록, 로그 |
| **변경 이력 데이터** | 데이터 내용 변경 시마다 이력 기록 | 주문 정보 변경 이력 |
| **진행 이력 데이터** | 업무 진행 과정에 따라 이력 기록 | 주문 처리 진행 상황 |

### 4.1 이력 관리 방식

| 방식 | 설명 | 특징 |
|------|------|------|
| 선분 이력 | 시작일~종료일로 유효 기간(선분) 표현 | 기간 조회에 유리, 데이터량 적음 |
| 점 이력 | 변경될 때마다 새 행으로 저장 | 변경 이력 추적에 유리, 데이터량 많음 |

**선분 이력 예시:**

```
고객ID | 등급 | 시작일     | 종료일
1001   | 일반 | 2024-01-01 | 2024-06-30
1001   | 우수 | 2024-07-01 | 9999-12-31 (현재)
```

**점 이력 예시:**

```
고객ID | 등급 | 변경일시
1001   | 일반 | 2024-01-01 09:00:00
1001   | 우수 | 2024-07-01 14:30:00
```

### 4.2 이력 관리 적용 기준

| 적용 기준 | 내용 |
|---------|------|
| 변경 빈도 | 자주 변경되는 중요 속성 |
| 조회 요건 | 과거 시점 데이터 조회 필요 |
| 감사 요건 | 데이터 변경 이력 감사 필요 |
| 법규 요건 | 데이터 보존 기간 법규 준수 |

## 5. 논리 데이터 모델 품질 검토

논리 모델 완성 후 품질 기준에 따라 검토한다.

### 5.1 데이터 모델 품질 기준

| 기준 | 정의 |
|------|------|
| 정확성(Accuracy) | 업무 현실을 올바르게 반영 |
| 완전성(Completeness) | 필요한 요소가 빠짐없이 정의됨 |
| 일관성(Consistency) | 전사 모델 전체에서 동일한 방식으로 정의 |
| 유연성(Flexibility) | 변화에 최소 수정으로 대응 가능 |
| 확장성(Extensibility) | 새 요건 추가 시 쉽게 확장 가능 |
| 통합성(Integration) | 전사 모델과 일관되게 통합 |

### 5.2 논리 모델 검토 체크리스트

| 항목 | 확인 내용 |
|------|---------|
| 엔터티 | 엔터티명 명확성, 필수 속성 포함 여부 |
| 속성 | 원자값 여부(1NF), 불필요한 중복 없음 |
| 식별자 | 유일성·최소성·불변성·존재성 만족 |
| 관계 | 업무 규칙과 일치, 카디널리티·참여도 정확 |
| 정규화 | 3NF 이상 정규화 수행 여부 |
| 이력 관리 | 필요한 엔터티에 이력 모델 적용 |
| 명명 규칙 | 표준 명명 규칙 준수 |

### 5.3 논리 모델 검토 방법

| 방법 | 설명 |
|------|------|
| 자동화 진단 | 데이터 모델 관리 도구를 통한 자동 오류 탐지 |
| 전문가 검토 | 데이터 아키텍트의 수동 검토 |
| 이해관계자 검토 | 현업 담당자와 함께 업무 일치성 확인 |

## 출제 포인트

1. 논리 데이터 모델링: 개념 모델 기반 속성·관계 상세화, DBMS 독립
2. 속성 유형: 기본·설계·파생 (파생 = 다른 속성으로 계산)
3. 도메인(Domain): 속성이 가질 수 있는 값의 범위·타입·형식 정의
4. 슈퍼타입/서브타입: 공통 속성을 슈퍼타입에, 특수 속성을 서브타입에
5. 배타적 서브타입: 하나의 서브타입에만 속함
6. 식별 관계: 부모 PK → 자식 PK 일부 구성 (실선)
7. 비식별 관계: 부모 PK → 자식 일반 FK (점선)
8. 선분 이력: 시작일~종료일로 기간 표현, 기간 조회에 유리
9. 점 이력: 변경 시마다 새 행 추가, 변경 이력 추적에 유리
10. 데이터 모델 품질 6기준: 정확성·완전성·일관성·유연성·확장성·통합성
11. 논리 모델 검토: 자동화 진단·전문가 검토·이해관계자 검토
