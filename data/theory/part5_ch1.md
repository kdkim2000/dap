# 5과목 1장: 데이터베이스 설계

## 1. 저장공간 설계

**저장공간 설계**는 데이터베이스의 물리적 저장 구조를 결정하는 작업으로, 성능·가용성·관리 용이성을 고려하여 설계한다.

### 1.1 테이블스페이스(Tablespace)

| 구분 | 설명 |
|------|------|
| 테이블스페이스 | 데이터 파일의 논리적 그룹, 테이블·인덱스 등을 저장하는 단위 |
| 데이터 파일 | 실제 OS 파일, 테이블스페이스를 구성 |
| 세그먼트 | 테이블스페이스 내 객체(테이블·인덱스) 저장 공간 |
| 익스텐트 | 연속된 데이터 블록의 집합 |
| 데이터 블록 | 최소 I/O 단위 (8KB, 16KB 등) |

**테이블스페이스 유형**:
- **시스템 테이블스페이스**: 데이터 딕셔너리, 시스템 객체
- **언두 테이블스페이스**: 트랜잭션 롤백용 언두 데이터
- **임시 테이블스페이스**: 정렬, 해시 작업 임시 공간
- **사용자 테이블스페이스**: 업무 데이터 저장

### 1.2 파티셔닝(Partitioning)

대용량 테이블을 논리적으로 분할하여 성능·관리성을 향상시키는 기법.

| 파티셔닝 유형 | 설명 | 적합 사례 |
|------|------|------|
| 범위 분할(Range) | 컬럼 값 범위로 분할 | 날짜 기반 데이터 |
| 목록 분할(List) | 특정 값 목록으로 분할 | 지역·부서 코드 |
| 해시 분할(Hash) | 해시 함수 결과로 분할 | 균등 분산 |
| 복합 분할(Composite) | 두 가지 이상 방법 조합 | 대용량 복합 조건 |

> **핵심**: 파티셔닝을 통해 파티션 프루닝(Partition Pruning)이 가능하며, 쿼리 시 불필요한 파티션을 제외하여 성능을 향상시킨다.

## 2. 무결성 설계

**무결성**은 데이터의 정확성·일관성·유효성을 보장하기 위한 규칙이다.

### 2.1 무결성 유형

| 무결성 유형 | 설명 | 구현 방법 |
|------|------|------|
| 개체 무결성 | 기본키는 NULL 불가, 유일해야 함 | PRIMARY KEY |
| 참조 무결성 | 외래키는 참조 테이블의 PK에 존재해야 함 | FOREIGN KEY |
| 도메인 무결성 | 컬럼 값이 허용된 도메인(범위) 내에 있어야 함 | CHECK, DEFAULT, NOT NULL |
| 사용자 정의 무결성 | 업무 규칙에 따른 제약조건 | TRIGGER, 애플리케이션 로직 |

### 2.2 참조 무결성 옵션

```sql
FOREIGN KEY (dept_id) REFERENCES department(dept_id)
  ON DELETE CASCADE     -- 부모 삭제 시 자식도 삭제
  ON DELETE SET NULL    -- 부모 삭제 시 자식 FK를 NULL로
  ON DELETE RESTRICT    -- 자식이 있으면 부모 삭제 금지
  ON DELETE NO ACTION   -- RESTRICT와 유사 (기본값)
```

### 2.3 제약조건 설계 원칙

1. PRIMARY KEY: 모든 테이블에 반드시 지정
2. FOREIGN KEY: 참조 무결성 자동 보장
3. NOT NULL: 필수 입력 컬럼에 지정
4. UNIQUE: 중복 불가 컬럼에 지정
5. CHECK: 도메인 값 검증

## 3. 인덱스 설계

**인덱스**는 데이터 검색 속도를 향상시키는 자료구조. 조회 성능을 높이지만 DML 성능을 저하시킬 수 있다.

### 3.1 인덱스 유형

| 유형 | 구조 | 특징 | 사용 상황 |
|------|------|------|------|
| B-Tree 인덱스 | 균형 이진 트리 | 범위 검색, 정렬에 효과적 | 대부분의 경우 |
| 비트맵 인덱스 | 비트 배열 | 중복도 높은 컬럼, DW 환경 | 성별·상태코드 |
| 함수기반 인덱스 | 컬럼 표현식 | 함수·수식을 인덱스화 | `UPPER(name)` 검색 |
| 클러스터드 인덱스 | 데이터 정렬 저장 | 범위 검색 최적 | PK 인덱스(SQL Server) |
| Non-Clustered | 별도 인덱스 구조 | 데이터와 분리 | 보조 인덱스 |
| 해시 인덱스 | 해시 테이블 | 등호 검색에 최적 | 메모리 테이블 |

### 3.2 인덱스 설계 기준

- **선택도(Selectivity)가 높은 컬럼**: 컬럼 값이 다양할수록 효과적
- **WHERE 조건 컬럼**: 자주 조회 조건으로 사용되는 컬럼
- **JOIN 조건 컬럼**: FK 컬럼에 인덱스 권장
- **ORDER BY, GROUP BY 컬럼**: 정렬 성능 향상
- **인덱스 남발 지양**: 테이블당 인덱스 수가 많으면 DML 성능 저하

> **핵심**: 인덱스는 SELECT 성능을 높이지만, INSERT/UPDATE/DELETE 시 인덱스 유지 비용이 발생한다.

### 3.3 클러스터드 vs 비클러스터드 인덱스 상세 비교

| 구분 | 클러스터드(Clustered) | 비클러스터드(Non-Clustered) |
|------|------|------|
| 데이터 저장 | 인덱스 순서대로 물리 정렬 저장 | 별도 인덱스 구조, 데이터는 분리 |
| 리프 노드 | 실제 데이터 페이지 | 행 포인터(ROWID 또는 PK) |
| 테이블당 개수 | **1개만** 가능 | 다수 생성 가능 |
| 범위 검색 | 매우 효율적 (물리 연속) | 비효율적 (페이지 점프) |
| 생성 기준 | 주로 PK | 보조 인덱스 |

### 3.4 복합 인덱스(Composite Index) 설계 원칙

- **선택도 높은 컬럼을 앞에 배치**: 첫 번째 컬럼이 WHERE 조건에 없으면 인덱스 미사용
- **자주 사용되는 조건 컬럼을 앞에 배치**
- **등호(=) 조건 컬럼 → 범위(<,>) 조건 컬럼 순서**

```sql
-- 좋은 예: 선택도 높은 컬럼(dept_id=) 앞, 범위(salary BETWEEN) 뒤
CREATE INDEX idx_emp ON TB_EMPLOYEE(dept_id, salary);
WHERE dept_id = 10 AND salary BETWEEN 3000000 AND 5000000
```

> **결합 인덱스 핵심**: 항상 '='(Equal) 조건으로 사용되는 컬럼을 선두에 배치. 선두 컬럼 없으면 인덱스 미사용.

### 3.5 인덱스 아키텍처 비교 및 특수 기법

#### B-Tree vs 비트맵 인덱스 상세 비교

| 구분 | B-Tree 인덱스 | 비트맵(Bitmap) 인덱스 |
|------|------|------|
| 구조 | 루트-브랜치-리프 노드 (균형 트리) | 컬럼값마다 비트 배열(0/1) 구성 |
| 적합 환경 | OLTP (데이터 변경 빈번) | DW/DSS (조회 위주) |
| 적합 컬럼 | Cardinality 높은 컬럼 (사원번호, 이름) | Cardinality 낮은 컬럼 (성별, 상태코드) |
| 탐색 깊이 | 어떤 값이든 동일한 탐색 깊이 | 비트 AND/OR 연산으로 빠른 집합 처리 |
| DML 성능 | 행 레벨 락 → DML 성능 양호 | **블록 레벨 락 → 동시 DML 성능 저하** |
| 부정/OR 조건 | 인덱스 미활용 가능 | **Not, OR 조건에서도 인덱스 활용** |

#### 특수 인덱스 활용 기법

| 기법 | 설명 | 사용 상황 |
|------|------|------|
| **함수기반 인덱스(FBI)** | 함수·수식 결과값을 인덱스화 | `WHERE UPPER(name) = 'KIM'` 검색 |
| **인덱스 스킵 스캔** | 결합 인덱스 선두 컬럼 조건 없어도 탐색 | 선두 컬럼 종류가 적을 때 (성별 등) |
| **비트맵 조인 인덱스** | 조인 연결 고리를 비트맵 인덱스화 | DW에서 팩트-차원 테이블 조인 성능 향상 |

- **FBI**: 원시 데이터가 아닌 연산 결과를 미리 인덱스화 → 함수가 적용된 WHERE 조건에서 인덱스 활용 가능
- **스킵 스캔**: 선두 컬럼(성별: 남/여)의 종류가 적으면, 선두 값을 하나씩 건너뛰며 리프 블록 탐색
- **비트맵 조인 인덱스**: 테이블 접근 없이 인덱스만으로 조인 결과 도출 → DW 집계 쿼리 성능 대폭 향상

## 4. 분산 설계

**분산 데이터베이스**는 여러 물리적 위치에 데이터를 저장·관리하는 시스템.

### 4.1 수평 분할(Horizontal Partitioning)

테이블의 **행(Row)** 을 기준으로 분할. 파티셔닝과 유사.

```
TB_SALES (전체)
  → TB_SALES_2024 (2024년 데이터)
  → TB_SALES_2025 (2025년 데이터)
```

### 4.2 수직 분할(Vertical Partitioning)

테이블의 **열(Column)** 을 기준으로 분할. 자주 사용하는 컬럼과 드문 컬럼을 분리.

```
TB_EMPLOYEE (전체)
  → TB_EMPLOYEE_BASIC (사번, 이름, 부서) — 자주 사용
  → TB_EMPLOYEE_DETAIL (주소, 가족정보, 사진) — 드물게 사용
```

### 4.3 복제(Replication)

동일 데이터를 여러 서버에 복사하여 가용성·읽기 성능 향상.

| 복제 유형 | 설명 |
|------|------|
| 마스터-슬레이브 | 마스터에서 쓰기, 슬레이브에서 읽기 분산 |
| 멀티-마스터 | 여러 서버에서 쓰기 가능, 충돌 해결 필요 |
| 피어-투-피어 | 모든 노드가 동등, 분산 트랜잭션 필요 |

### 4.4 RAID(Redundant Array of Independent Disks)

여러 디스크를 묶어 성능과 안전성을 향상시키는 기술.

| RAID 레벨 | 방식 | 성능 | 안전성 | 유효 용량 | 특징 |
|------|------|:---:|:---:|:---:|------|
| **RAID 0** | 스트라이핑(분산 저장) | ↑↑ | ↓↓ | 100% | 1개 고장 시 전체 손실 |
| **RAID 1** | 미러링(복사 저장) | 보통 | ↑↑ | 50% | 하나 고장해도 복구 가능 |
| **RAID 5** | 스트라이핑 + 패리티 분산 | ↑ | ↑ | (n-1)/n | 디스크 1개 고장 복구 가능 |
| **RAID 6** | 스트라이핑 + 패리티 2개 | 보통 | ↑↑ | (n-2)/n | 디스크 2개 고장 복구 가능 |
| **RAID 10** | RAID 1 + RAID 0 조합 | ↑↑ | ↑↑ | 50% | 고성능·고가용성, 비용↑ |

> **핵심**: RAID 5는 패리티를 분산 저장하여 성능과 안전성의 균형. RAID 10은 미러링+스트라이핑으로 최고 성능·안전성이나 비용 높음.

### 4.5 CAP 정리 (분산 시스템)

분산 시스템에서 3가지 특성을 동시에 모두 만족시킬 수 없다는 이론.

| 특성 | 설명 |
|------|------|
| 일관성(Consistency) | 모든 노드가 동일한 데이터를 반환 |
| 가용성(Availability) | 요청에 항상 응답 (정상 응답) |
| 분할 허용성(Partition Tolerance) | 네트워크 분할 발생 시에도 동작 |

- **CP 시스템**: 일관성 + 분할 허용 (가용성 희생) → HBase, Zookeeper
- **AP 시스템**: 가용성 + 분할 허용 (일관성 희생) → Cassandra, DynamoDB

### 4.6 분산 투명성 (5대 투명성)

사용자가 데이터의 분산 형태를 알 필요 없이 하나의 DB처럼 이용하도록 보장하는 원칙.

| 투명성 유형 | 설명 |
|------|------|
| **분할 투명성** | 데이터가 어떻게 나뉘어 있는지 알 필요 없음 |
| **위치 투명성** | 물리적 데이터 저장 위치를 알 필요 없음 |
| **중복(복제) 투명성** | 데이터가 어디에 복제되어 있는지 알 필요 없음 |
| **장애 투명성** | 특정 지역 DB 장애 발생 시 자기 지역 DB로 정상 업무 지속 |
| **병행 투명성** | 다수 트랜잭션 동시 접근 시 일관성 보장 — **2단계 커밋(2-Phase Commit)** 기술 활용 |

> **핵심**: DAP 시험에서 "분산 DB의 5대 투명성"으로 자주 출제. 분할·위치·중복·장애·병행 5가지 암기.

## 5. 보안 설계

### 5.1 접근 제어

| 방법 | 설명 |
|------|------|
| DAC(임의 접근 제어) | 데이터 소유자가 접근 권한 부여 |
| MAC(강제 접근 제어) | 시스템이 보안 레벨로 제어 |
| RBAC(역할 기반 접근 제어) | 역할(Role)에 권한을 부여, 사용자에 역할 할당 |

```sql
-- RBAC 예시
GRANT SELECT, INSERT ON TB_EMPLOYEE TO role_hr;
REVOKE DELETE ON TB_EMPLOYEE FROM role_hr;
GRANT role_hr TO user_kim;
```

### 5.2 뷰(View)를 통한 보안

민감 컬럼을 제외한 뷰 생성으로 데이터 보안.

```sql
CREATE VIEW V_EMPLOYEE_PUBLIC AS
SELECT emp_id, emp_name, dept_id  -- 급여·주민번호 제외
FROM TB_EMPLOYEE;
```

### 5.3 세부 접근통제 조건

기본 접근 제어(신분/역할 기반)에 더해 다중 조건을 적용하는 고급 보안 기법.

| 조건 유형 | 설명 | 예시 |
|------|------|------|
| **값에 의한 종속 통제** | 데이터 값이 특정 기준을 충족할 때만 접근 허용 | 급여 500만원 이상 데이터는 임원만 조회 |
| **다중 주체 통제** | 다수의 사용자가 동시에 인증해야 접근 가능 | 핵심 기밀 데이터는 두 명 이상 동시 승인 |
| **컨텍스트 기반 통제** | 접근 시간·위치·네트워크 등 환경 분석 후 허용 | 근무 시간 외·사내망 외 접근 차단 |

### 5.4 감사 추적 (Audit Trail)

인가·비인가 사용자 모두를 대상으로 **언제·누가·어떤 데이터를·어떤 조작을 했는지** 기록을 남겨 추적하는 기능.

```sql
-- Oracle 감사 설정 예시
AUDIT SELECT ON TB_SALARY BY ACCESS;      -- 급여 테이블 조회 감사
AUDIT INSERT, UPDATE, DELETE ON TB_EMPLOYEE BY SESSION;  -- 변경 감사
```

| 감사 항목 | 설명 |
|------|------|
| 로그인/로그아웃 기록 | 접속 시각·IP·계정 |
| DDL 감사 | 테이블 생성·변경·삭제 이력 |
| DML 감사 | 민감 테이블 조회·변경 이력 |
| 권한 부여/회수 감사 | GRANT/REVOKE 이력 |

> **핵심**: 사고 발생 시 원인 재구성·책임성 확보를 위해 필수. 감사 추적 로그는 별도 안전한 저장소에 보관.

### 5.5 암호화

| 암호화 유형 | 알고리즘 | 용도 |
|------|------|------|
| 대칭키 암호화 | AES-128/256 | 저장 데이터 암호화 (속도 빠름) |
| 비대칭키 암호화 | RSA, ECC | 키 교환, 전자서명 |
| 해시 함수 | SHA-256, bcrypt | 패스워드 저장 (복호화 불가) |
| 전송 암호화 | TLS 1.2+/SSL | 네트워크 전송 구간 보호 |

**컬럼 레벨 암호화 적용 대상**:
- 주민등록번호: AES-256으로 암호화 저장
- 카드번호: 마스킹(앞4자리+*+뒤4자리) 또는 암호화
- 패스워드: bcrypt 단방향 해시 (솔트 포함)
- 계좌번호: 암호화 저장

> **핵심**: 패스워드는 복호화가 필요 없으므로 단방향 해시(bcrypt, SHA-256)를 사용. 개인정보는 AES-256 대칭키 암호화.

## 출제 포인트

1. **무결성 4가지**: 개체·참조·도메인·사용자 정의 무결성
2. **참조 무결성 옵션**: CASCADE / SET NULL / RESTRICT / NO ACTION
3. **B-Tree vs 비트맵 인덱스**: B-Tree는 OLTP·Cardinality 높은 컬럼, 비트맵은 DW·중복도 높은 컬럼·NOT/OR 조건 활용
4. **비트맵 DML 주의**: 블록 레벨 락 → 동시 DML 성능 저하, DW 환경에서만 사용
5. **특수 인덱스**: FBI(함수기반/수식 결과 인덱스화), 스킵스캔(선두 컬럼 없어도 탐색), 비트맵 조인(DW 조인 성능)
6. **결합 인덱스**: Equal(=) 조건 컬럼을 선두에, 범위(<,>) 조건 컬럼은 뒤에
7. **파티셔닝 유형**: 범위·목록·해시·복합
8. **인덱스 주의**: 많은 인덱스 → DML 성능 저하
9. **클러스터드 vs 비클러스터드**: 클러스터드=데이터 정렬 저장(테이블당 1개), 비클러스터드=별도 구조(다수 가능)
10. **수평 분할 vs 수직 분할**: 행 기준 vs 열 기준
11. **분산 투명성 5가지**: 분할·위치·중복(복제)·장애·병행 — 5가지 전부 암기 필수
12. **병행 투명성**: 2단계 커밋(2-Phase Commit)으로 분산 DB 간 일관성 보장
13. **RAID 레벨**: RAID 0(스트라이핑/성능↑안전↓), RAID 1(미러링/안전↑), RAID 5(패리티분산/균형), RAID 10(고성능·고안전)
14. **CAP 정리**: 일관성·가용성·분할허용성 — 3가지 동시 만족 불가
15. **RBAC**: 역할에 권한 부여, 사용자에 역할 할당
16. **세부 접근통제**: 값에 의한 종속/다중 주체/컨텍스트 기반 통제
17. **감사 추적(Audit Trail)**: 언제·누가·어떤 데이터를·어떤 조작을 했는지 기록 — 사고 원인 재구성·책임성 확보
18. **암호화 알고리즘**: AES(대칭키/저장), TLS(전송), bcrypt/SHA-256(패스워드 해시)
