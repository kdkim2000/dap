# 3과목 2장: 데이터 표준 수립

## 1. 데이터 명칭 표준

데이터 명칭 표준은 조직 내 데이터 항목의 이름을 일관되게 부여하기 위한 규칙 체계이다.

> **핵심**: 데이터 명칭 표준은 단어 표준 → 도메인 표준 → 용어 표준의 계층적 구조로 구성되며, 이를 조합하여 일관된 데이터 항목명을 생성한다.

### 단어 표준(Word Standard)

- **정의**: 데이터 항목명에 사용되는 개별 단어(최소 의미 단위)의 표준
- **구성**: 한글명, 영문명, 약어, 설명
- **목적**: 임의 약어 사용 방지, 일관된 영문명 부여

| 단어(한글) | 영문명 | 표준 약어 | 설명 |
|---------|------|--------|-----|
| 고객 | CUSTOMER | CUST | 상품·서비스를 구매하는 주체 |
| 번호 | NUMBER | NO | 식별을 위한 고유 번호 |
| 금액 | AMOUNT | AMT | 화폐 단위의 수치 |
| 날짜 | DATE | DT | 특정 일자 |
| 이름 | NAME | NM | 사람·사물의 호칭 |
| 코드 | CODE | CD | 표준화된 분류 값 |

### 도메인 표준(Domain Standard)

- **정의**: 데이터 항목이 가질 수 있는 값의 범위·형식·유형을 정의한 표준
- **목적**: 동일한 성격의 데이터에 동일한 물리적 형식 적용

| 도메인명 | 데이터유형 | 길이/정밀도 | 설명 |
|--------|---------|-----------|-----|
| 일반번호 | VARCHAR | 20 | 일반적인 번호 유형 |
| 금액 | DECIMAL | 15,2 | 화폐 금액 |
| 일반날짜 | DATE | - | 표준 날짜 (YYYY-MM-DD) |
| 일반코드 | VARCHAR | 10 | 분류 코드 |
| 이메일주소 | VARCHAR | 100 | 이메일 형식 |
| 전화번호 | VARCHAR | 20 | 전화번호 형식 |
| 사용여부 | CHAR | 1 | Y/N 이진 값 |
| 장문텍스트 | VARCHAR | 4000 | 설명·비고 텍스트 |

### 용어 표준(Term Standard)

- **정의**: 비즈니스에서 사용하는 용어(단어의 조합)를 표준화한 것
- **구성**: 용어명(한글), 영문명, 용어 설명, 관련 용어
- **단어 표준과의 차이**: 단어 표준이 개별 단어라면, 용어 표준은 비즈니스 의미를 가진 합성 용어

| 용어(한글) | 영문명 | 도메인 | 설명 |
|---------|------|------|-----|
| 고객번호 | CUSTOMER_NO | 일반번호 | 고객 식별 고유번호 |
| 주문금액 | ORDER_AMOUNT | 금액 | 주문의 총 금액 |
| 가입일자 | JOIN_DATE | 일반날짜 | 서비스 가입 날짜 |
| 상품코드 | PRODUCT_CODE | 일반코드 | 상품 식별 코드 |

> **핵심**: 단어 표준(원자 단위) → 용어 표준(비즈니스 합성 단위) → 도메인 표준(물리적 형식)의 3계층이 데이터 명칭 표준의 핵심이다.

## 2. 데이터 형식 표준

### 데이터 유형 표준

각 데이터 유형별로 사용 기준을 표준화:

| 데이터 유형 | 사용 기준 | 예시 |
|---------|--------|-----|
| VARCHAR | 가변 길이 문자열 | 이름, 주소, 설명 |
| CHAR | 고정 길이 코드값 | 성별코드(M/F), 사용여부(Y/N) |
| INTEGER | 정수 | 수량, 순번 |
| DECIMAL(p,s) | 소수점 있는 수치 | 금액, 비율 |
| DATE | 날짜 (시간 없음) | 가입일, 생년월일 |
| TIMESTAMP | 날짜+시간 | 거래일시, 수정일시 |
| CLOB | 대용량 텍스트 | 상품 상세 설명 |
| BLOB | 대용량 바이너리 | 첨부 파일 |

### 날짜·시간 형식 표준

```
날짜: YYYY-MM-DD (예: 2025-01-15)
시간: HH:MM:SS (예: 14:30:00)
날짜+시간: YYYY-MM-DD HH:MM:SS (예: 2025-01-15 14:30:00)
연도: YYYY (예: 2025)
연월: YYYY-MM (예: 2025-01)
```

### 수치 형식 표준

```
금액: 소수점 2자리, 천 단위 구분자 없음 (예: 1500000.00)
비율: 소수점 2자리, 백분율 (예: 15.75)
좌표: 소수점 6자리 (예: 37.566535)
```

## 3. 코드 표준

### 코드 정의(Code Definition)

코드(Code)는 특정 도메인의 값을 약속된 기호·숫자로 표현한 것으로, 데이터의 표준화와 효율화를 위해 사용된다.

#### 코드의 특성

| 특성 | 설명 |
|-----|------|
| 유일성 | 하나의 코드값은 하나의 의미만 가짐 |
| 포괄성 | 해당 도메인의 모든 가능한 값을 포함 |
| 안정성 | 코드값이 자주 변경되지 않음 |
| 간결성 | 코드값은 최대한 간결하게 |

### 코드 설계(Code Design)

#### 코드 체계 유형

| 코드 유형 | 설명 | 예시 |
|--------|-----|------|
| 순번 코드 | 단순 일련번호 | 001, 002, 003... |
| 약자 코드 | 의미 있는 약자 | M(남성), F(여성) |
| 분류 코드 | 계층적 분류 체계 | 1=식품, 11=과일, 111=열대과일 |
| 블록 코드 | 그룹별 번호 범위 | 100-199: 한국, 200-299: 미국 |
| 복합 코드 | 여러 코드의 조합 | 지역코드 + 구분코드 |

#### 코드 설계 원칙

1. 확장성: 새로운 코드값 추가를 고려한 여유 공간
2. 유추 가능성: 코드값에서 의미를 일부 유추 가능하면 좋음
3. 비어있는 범위: 미래 추가를 위한 예약 범위 설정

### 코드 관리(Code Management)

```
코드 관리 절차:

1. 코드 등록 요청 (업무 담당자)
   ↓
2. 코드 검토 (데이터 스튜어드)
   - 기존 코드와 중복 여부 확인
   - 코드 설계 기준 적합성 검토
   ↓
3. 코드 승인 (데이터 관리자)
   ↓
4. 코드 사전 등록 (데이터 표준팀)
   ↓
5. 시스템 적용 (개발팀)
   ↓
6. 코드 변경/폐기 관리
   - 변경: 이력 관리, 영향 분석 필수
   - 폐기: 사용 여부 코드(Y/N)로 관리, 물리 삭제 지양
```

## 4. 메타데이터 표준

### 메타데이터의 정의

메타데이터(Metadata)란 "데이터에 대한 데이터"로, 데이터를 설명하고 관리하기 위한 구조화된 정보이다.

> **핵심**: 메타데이터는 기술적 메타데이터(물리적 구조), 관리적 메타데이터(관리 정보), 구조적 메타데이터(논리적 구조)의 3가지 유형으로 분류된다.

### 메타데이터 유형

#### 기술적 메타데이터(Technical Metadata)

- **정의**: 데이터의 물리적 저장 구조와 기술적 속성 정보
- **예시**:
  - 테이블명, 컬럼명, 데이터 유형, 길이
  - 인덱스, 파티션 정보
  - 저장 위치 (데이터베이스, 테이블스페이스)
  - 데이터 볼륨 (행 수, 크기)

#### 관리적 메타데이터(Administrative Metadata)

- **정의**: 데이터 자산의 관리와 거버넌스에 필요한 정보
- **예시**:
  - 데이터 오너 (부서, 담당자)
  - 생성일, 최종 수정일
  - 접근 권한 (보안 등급)
  - 보관 기간 및 폐기 규칙
  - 출처 (Source System)

#### 구조적 메타데이터(Structural Metadata)

- **정의**: 데이터 개체 간의 논리적 구조와 관계 정보
- **예시**:
  - 엔터티-관계 정보 (ERD)
  - 참조 무결성 관계 (PK-FK)
  - 비즈니스 규칙과 제약 조건
  - 데이터 계보(Data Lineage)

| 메타데이터 유형 | 핵심 내용 | 주요 사용자 |
|------------|---------|-----------|
| 기술적 | 물리적 구조, DB 정보 | DBA, 개발자 |
| 관리적 | 소유권, 보안, 이력 | 데이터 관리자, 감사 |
| 구조적 | 논리 구조, 관계, 계보 | 데이터 아키텍트, 분석가 |

### 메타데이터 관리 시스템(MDR: Metadata Registry)

- **기능**: 메타데이터 등록, 검색, 버전 관리
- **ISO 11179**: 메타데이터 레지스트리 국제 표준
- **구성 요소**: 데이터 요소 개념, 데이터 요소, 값 도메인, 표현 클래스

## 5. 데이터 표준화 산출물

### 주요 산출물 목록

| 산출물 | 내용 | 형식 |
|------|-----|------|
| 데이터 표준화 지침서 | 표준화 원칙, 명명 규칙 전반 | 문서 |
| 단어 사전 | 표준 단어 목록 (한글명, 영문명, 약어) | 엑셀/DB |
| 용어 사전 | 표준 용어 목록 (비즈니스 의미 포함) | 엑셀/DB |
| 도메인 사전 | 도메인 목록 (유형, 형식, 규칙) | 엑셀/DB |
| 코드 사전 | 코드 목록 및 코드값 정의 | 엑셀/DB |
| 데이터 사전 | 전체 데이터 항목 메타데이터 | DB |
| 표준 ERD | 개념/논리 데이터 모델 표준 | 다이어그램 |

### 데이터 표준화 산출물 간의 관계

```
단어 사전 ──→ (단어 조합) ──→ 용어 사전
                                   │
                          도메인 사전 ←─ 연결 ─→ 용어 사전
                                   │
                            데이터 사전 (용어 + 도메인 조합)
                                   │
                             코드 사전 ←─ 참조 ─→ 데이터 사전
```

## 6. 데이터 표준 정의 심화 (고려사항)

### 표준 단어 고려사항

1. 일반적인 업무용 단어 선정 (누구나 동일하게 이해)
2. 동의어 중 대표 단어 하나만 선정
3. 접두어·접미어 같은 한 자리 단어 배제
4. 높은 빈도로 조합되는 단어는 하나의 표준 단어로 정의 가능
5. 발음식 단어 지양 (영문명 작성 기준)
6. 한글명당 하나의 영문명만 허용 (동음이의어 불가)
7. 영문명당 복수의 한글명 허용 가능 (이음동의어)

### 표준 코드 고려사항

1. 표준기구·정부에서 정의한 표준 코드 우선 사용
2. 텍스트 직접 입력 최소화 (코드 체계로 대체)
3. 엄격한 기준으로 관리 (사용자 임의 수정 금지)
4. 코드 값 변경 시 기존 코드는 사용 중지 처리
5. **표준 코드 - 현행 코드 변환 매핑 정보** 반드시 기록
6. 코드 통합 대상 파악 및 단일화 작업 수행

### 담당자별 표준 관리 업무 역할

| 담당자 | 담당 업무 |
|--------|---------|
| 데이터 관리자 | 표준 단어·용어·도메인·코드 신규 생성 및 변경 요청 |
| 데이터베이스 관리자(DBA) | 신규·변경 사항 검토, 표준 준수 체크, 영향도 분석, 메타 DB 등록 |
| 전사 데이터 관리자 | 변경 사항 영향도 파악, 평가서 작성, 테스트 및 검증 |
| 자문 역할 | 전사 관점에서의 표준 가이드 제시 |

## 7. 출제 포인트

1. **명칭 표준 3계층**: 단어 표준(원자) → 용어 표준(합성) → 도메인 표준(형식)
2. **도메인 표준**: 동일 성격 데이터에 동일 물리 형식 적용, 예) 금액 도메인: DECIMAL(15,2)
3. **메타데이터 3가지 유형**: 기술적(물리구조), 관리적(소유권·보안), 구조적(논리구조·관계)
4. **코드 체계 유형**: 순번, 약자, 분류, 블록, 복합 코드
5. **코드 폐기 원칙**: 물리 삭제 대신 사용여부 코드(Y/N)로 관리
6. **ISO 11179**: 메타데이터 레지스트리 국제 표준
7. **단어 사전 구성**: 한글명, 영문명, 약어, 설명의 4요소
8. **CHAR vs VARCHAR**: CHAR는 고정 길이 코드값, VARCHAR는 가변 길이 문자열
9. **데이터 계보(Data Lineage)**: 구조적 메타데이터의 일종, 데이터 흐름·변환 이력
10. **표준화 산출물 체계**: 지침서 → 단어/용어/도메인/코드 사전 → 데이터 사전 → 표준 ERD
