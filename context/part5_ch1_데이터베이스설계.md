제공해주신 '31강 데이터베이스 설계' 동영상 스크립트를 바탕으로, 누락되는 내용이 없도록 각 핵심 챕터별로 구조화하여 정리해 드립니다.

### 1. 인덱스 설계 (Index Design)
데이터베이스의 접근 성능을 최적화하기 위해 인덱스를 설계하는 원리와 다양한 인덱스 아키텍처에 대해 설명합니다.

*   **접근 경로 수집 및 분포도 분석:** 인덱스 설계를 위해서는 먼저 조건으로 자주 들어오는 패턴과 접근 경로를 수집해야 합니다. 인덱스 후보를 결정할 때는 데이터의 분포도를 따져보아야 하며, 단일 컬럼뿐만 아니라 여러 컬럼이 결합했을 때의 **결합 분포도**도 함께 분석해야 합니다.
*   **결합 인덱스 (Composite Index) 구성:** 다수의 조건이 함께 사용될 때 결합 인덱스의 컬럼 순서를 결정하는 것은 매우 중요합니다. 주로 **항상 '=' (Equal) 조건으로 사용되는 컬럼이나, 분포도가 좋아 변별력이 높은 컬럼을 선두에 배치**하여야 인덱스의 효율을 높일 수 있습니다.
*   **주요 인덱스 아키텍처 비교:**
    *   **B-Tree (Balanced) 인덱스:** 루트, 브랜치, 리프 노드로 구성되어 어떤 값을 찾든 동일한 탐색 깊이와 속도를 보장합니다. 데이터의 종류(Cardinality)가 많고 분포도가 좋은 컬럼에 유리하며, 주로 데이터 변경(입력, 수정, 삭제)이 빈번한 **OLTP 업무에 적합**합니다.
    *   **비트맵 (Bitmap) 인덱스:** 데이터를 0과 1의 비트로 표현하여 인덱스 크기를 획기적으로 줄인 구조입니다. 성별처럼 데이터 종류가 적은 경우 유리하며, B-Tree에서 처리하기 힘든 **부정형 비교(Not)나 OR 조건에서도 인덱스를 활용**할 수 있습니다. 다만 데이터 변경 시 블록 레벨의 락킹(Locking)이 발생하여 부하가 크므로, 주로 조회가 많은 **데이터 웨어하우스(DW) 환경에 적합**합니다.
*   **특수 인덱스 활용 기법:**
    *   **함수 기반 인덱스 (Function Based Index):** 원시 데이터를 곱하기 등 함수로 가공하여 조건으로 검색할 때, 그 연산 된 결과값을 미리 인덱스로 생성해두어 탐색을 빠르게 하는 기법입니다.
    *   **인덱스 스킵 스캔 (Index Skip Scan):** 결합 인덱스의 선두 컬럼이 조건에 주어지지 않더라도, 선두 컬럼의 종류가 많지 않다면 이를 건너뛰며(Skip) 해당 조건에 맞는 리프 블록만 부분적으로 탐색하는 방식입니다.
    *   **비트맵 조인 인덱스:** 특정 테이블과 조인하기 위해 필요한 연결 고리를 미리 비트맵 인덱스로 구성해 두고, 테이블 접근 없이 인덱스만으로 빠르게 조인 결과를 도출하는 기법입니다.

### 2. 분산 데이터베이스 설계 (Distributed Database Design)
논리적으로는 통합된 하나의 데이터베이스처럼 보이지만, 물리적으로는 네트워크 상에 분산된 시스템을 설계하는 방법론입니다.

*   **분산 데이터베이스의 5대 투명성(Transparency):** 사용자가 데이터의 분산 형태를 알 필요 없이 하나의 데이터베이스처럼 이용하도록 보장하는 원칙입니다.
    *   **분할 투명성:** 데이터가 어떻게 나뉘어 있는지 알 필요가 없음.
    *   **위치 투명성:** 물리적인 데이터 저장 위치를 알 필요가 없음.
    *   **중복 투명성:** 데이터가 어디에 중복(복제)되어 있는지 알 필요가 없음.
    *   **장애 투명성:** 특정 지역의 데이터베이스 장애가 발생해도, 자기 지역의 DB만 살아있다면 정상적으로 업무를 지속할 수 있음.
    *   **병행 투명성:** 다수 트랜잭션 시 데이터 일관성을 보장하며, 주로 분산된 DB 간의 업데이트를 동시에 성공하거나 실패하게 만드는 **2단계 커밋(2-Phase Commit)** 기술이 사용됨.
*   **테이블 분할 원칙:** 글로벌 데이터를 누락 없이 분할하는 **'완전성'**, 조인을 통해 원본으로 통합할 수 있는 **'재구성'**, 각 분할 간 데이터가 겹치지 않는 **'상호 중첩 배제'** 원칙을 따라야 합니다.
*   **분산 전략 및 복제:**
    *   **할당:** 테이블을 수평 혹은 수직으로 분할한 후, 업무 의존성을 고려하여 단일 노드 또는 중복 노드에 할당합니다. 
    *   **복제 (Replication):** 성능과 장애 대비를 위해 데이터를 다른 지역 서버에 복제합니다. 필요에 따라 실시간, 주기적 일괄 복제, 단순 읽기 전용(스냅샷) 또는 업데이트가 가능한 마스터 복제 등 다양한 방식이 활용됩니다.

### 3. 데이터베이스 보안 설계 (Security Design)
데이터베이스에 저장된 중요 자산을 인가되지 않은 자의 접근, 유출, 변조로부터 보호하기 위한 체계입니다.

*   **접근 통제 (Access Control) 기능:**
    *   **임의적 접근통제 (DAC):** 사용자의 신원(ID)에 근거하여 그랜트(Grant) 및 리보크(Revoke) 명령으로 권한을 부여하고 취소하는 방식입니다.
    *   **강제적 접근통제 (MAC):** 주체(사용자)와 객체(데이터)에 보안 등급을 매겨, 자신보다 높은 등급의 정보는 일체 접근할 수 없도록 강제하는 방식입니다.
*   **접근 통제 정책 및 모델:**
    *   **통제 정책:** 개인/그룹 단위의 **신분 기반 정책**, 보안 등급에 따른 **규칙 기반 정책**, 개인의 직무나 직책에 따라 권한을 부여하는 **역할 기반 정책(RBAC)**으로 나뉩니다.
    *   보안 접근을 설계하기 위해 사용자(주체), 데이터(객체), 그리고 권한(규칙)을 교차하여 정리하는 **접근 제어 행렬 (Access Control Matrix)**을 활용합니다.
*   **세부 접근 통제 조건:** 특정 금액 등 **값에 의한 종속 통제**, 다수의 사용자가 동시에 인증해야 하는 **다중 주체 통제**, 시간이나 네트워크 위치 등을 분석하는 **컨텍스트 기반 통제** 등을 활용하여 다중으로 보안을 적용할 수 있습니다.
*   **감사 추적 (Audit Trail):** 인가된 사용자든 뚫고 들어온 비인가 사용자든, 시스템 내에서 언제 누가 어떤 데이터를 실행하고 변경했는지 일련의 기록을 남겨 추적하는 필수 기능입니다. 이를 통해 사고 발생 시 원인을 재구성하고 책임성을 명확히 할 수 있습니다.
