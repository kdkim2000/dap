제공해주신 '32강 데이터베이스 이용' 동영상 스크립트의 내용을 누락 없이 각 핵심 챕터별로 구조화하여 정리해 드립니다.

### 1. 데이터베이스 관리 시스템(DBMS) 아키텍처 및 구조
데이터베이스가 기동되고 운영되는 데 필요한 전반적인 아키텍처(메모리, 프로세스, 저장 구조)를 설명합니다.

*   **데이터베이스 시작과 종료:**
    *   **시작(Startup) 3단계:** 초기화 매개변수 파일을 읽어 인스턴스를 기동하고(메모리 할당 및 프로세스 생성) -> 컨트롤 파일과 데이터/로그 파일을 점검하여 데이터베이스를 마운트(Mount)한 후 -> 일반 사용자가 접근 가능하도록 데이터베이스를 오픈(Open)합니다. 오픈 시점에 미확정된 트랜잭션에 대한 자동 복구(롤백 및 커밋)가 이루어집니다.
    *   **종료(Shutdown) 3단계:** 데이터베이스를 닫고(Close) -> 마운트를 해제한 뒤 -> 인스턴스를 종료하여 할당된 메모리와 프로세스를 반환합니다.
*   **데이터 딕셔너리 (Data Dictionary):** 데이터베이스 내의 모든 객체(테이블, 인덱스 등) 구조, 할당된 영역, 제약 조건, 사용자 권한 등의 메타데이터를 저장하는 읽기 전용 사전입니다.
*   **물리적/논리적 저장 구조:**
    *   **블록(Block):** 디스크 I/O의 최소 단위로, 업무 특성(OLTP/DW)에 따라 사이즈를 다르게 설정할 수 있습니다. 데이터 업데이트로 인해 행 크기가 늘어나 다른 블록에 데이터가 나뉘어 저장되는 **블록 체이닝/마이그레이션 현상을 방지하기 위해 빈 공간(PCTFREE)을 적절히 예약**해야 합니다.
    *   **익스텐트(Extent)와 세그먼트(Segment):** 연속된 블록의 모음이 익스텐트이며, 익스텐트들이 모여 테이블이나 인덱스와 같은 논리적 저장 공간인 세그먼트(테이블스페이스의 하위 단위)를 이룹니다.
*   **메모리 및 프로세스 구조:**
    *   **공유 메모리(SGA):** 디스크 I/O를 줄이기 위해 자주 쓰는 데이터를 캐싱하는 **DB 버퍼 캐시**, SQL 파싱 정보와 실행 계획을 공유하는 **공유 풀(Shared Pool)**, 복구를 위한 변경 내역을 저장하는 **로그 버퍼** 등으로 구성됩니다.
    *   **프로세스:** 사용자의 접속을 관리하는 세션 및 서버 프로세스와 DBWR, LGWR, PMON, SMON처럼 백그라운드에서 주기적으로 동작하며 메모리와 파일 간 동기화를 처리하는 백그라운드 프로세스가 있습니다.

### 2. 데이터 액세스와 처리 (SQL, 프로시저, 트리거)
데이터베이스 내부에서 데이터를 조회하고 조작하는 통로와 처리 엔진의 역할에 대해 다룹니다.

*   **옵티마이저 (Optimizer):** 사용자가 던진 SQL문의 실행 계획을 수립하는 핵심 엔진입니다.
    *   사전 정의된 규칙을 따르는 규칙 기반(Rule-Based)과, 데이터의 실제 분포도와 통계 정보를 바탕으로 최저 비용을 산출하는 **비용 기반(Cost-Based) 옵티마이저**로 나뉩니다. 인간의 개입이 필요할 때는 **힌트(Hint)**를 주어 최적의 실행 경로를 유도할 수 있습니다.
*   **SQL 수행 단계와 파싱 부하:** 문법을 검사하고 실행 계획을 세우는 '파싱' 단계, 처리 방법을 생성하는 단계, 실제 실행 단계로 나뉩니다. 이때 매번 새로운 실행 계획을 만드는 **하드 파싱(Hard Parsing)은 시스템 부하가 매우 크므로**, 한 번 만든 실행 계획을 재사용하는 **소프트 파싱(Soft Parsing)**을 유도해야 합니다.
*   **명령어 종류:**
    *   **DDL:** 테이블 등 구조를 정의하며(CREATE, DROP, ALTER) 암시적으로 커밋을 수반합니다.
    *   **DML:** 데이터를 조작하며(INSERT, UPDATE, DELETE, SELECT) 트랜잭션 제어(명시적 커밋)가 필요합니다.
    *   **DCL/TCL:** 트랜잭션(COMMIT, ROLLBACK)과 세션을 제어합니다.
*   **스토어드 프로시저 (Stored Procedure):** 일련의 처리 로직을 DB 내부에 미리 컴파일하여 저장해 두는 방식입니다. 네트워크 트래픽 감소, 파싱 부하 최소화, 보안 향상, 높은 재사용성의 장점이 있으며, 설계 시 **높은 응집도와 낮은 결합도**를 유지해야 합니다.
*   **트리거 (Trigger):** 테이블에 특정 DML(INSERT, UPDATE, DELETE) 이벤트가 발생할 때 연쇄적으로 지정된 작업을 자동 수행하는 객체입니다. 복잡한 업무 규칙 강제, 복제, 감사 추적 등에 유용하지만, 남용하면 유지보수가 어려워지고 예상치 못한 성능 저하(연쇄 발동)를 겪을 수 있습니다.

### 3. 트랜잭션, 동시성 제어 및 복구
다중 사용자 환경에서 데이터의 무결성과 일관성을 유지하고, 장애 발생 시 대처하는 방법입니다.

*   **트랜잭션 (Transaction) 특성:** 데이터의 일관성을 보장하기 위한 최소 작업 단위로, 원자성(All or Nothing), 일관성, 고립성(격리성), 영속성을 만족해야 합니다.
*   **고립화 수준 (Isolation Level):** 여러 트랜잭션이 동시에 실행될 때, 다른 트랜잭션이 변경 중인 데이터를 읽어 발생하는 문제(Dirty Read, Non-Repeatable Read, Phantom Read)를 방지하기 위해 0~3단계의 고립화 수준(Read Uncommitted ~ Serializable)을 적용합니다.
*   **동시성 제어와 락킹 (Locking):**
    *   **비관적 동시성 제어:** 다른 사용자가 동시에 수정할 수 있다고 가정하여, 데이터 조회 시점부터 강제로 락(잠금)을 거는 방식입니다 (`SELECT FOR UPDATE` 등).
    *   **락 에스컬레이션:** 행(Row) 단위 락이 너무 많이 발생하여 시스템 관리가 무거워지면, 페이지나 테이블 단위로 잠금 범위를 높여 부하를 줄이는 기술입니다.
    *   **교착 상태 (Deadlock):** 둘 이상의 프로세스가 서로 상대방이 점유한 자원을 무한정 기다리는 현상입니다. 락 유지 시간을 최소화하고, `NOWAIT` 옵션이나 시퀀스 객체를 활용하여 교착 상태를 예방해야 합니다.
*   **분산 트랜잭션 (2-Phase Commit):** 네트워크로 분산된 노드 간의 트랜잭션을 처리할 때, 준비 단계와 커밋 단계를 거쳐 모든 노드가 완벽히 성공하거나 롤백되도록 하여 글로벌 일관성을 지킵니다.
*   **데이터베이스 복구 (Recovery) 및 백업:**
    *   비정상적인 장애 발생 시 이전 상태로 되돌리거나, 변경된 시점까지 다시 재생성하기 위해 **리두 로그(Redo Log) 파일**을 사용합니다. DB 버퍼 변경 내역은 성능을 위해 일정 부분 동기/비동기적으로 로그 파일에 쓰입니다.
    *   업무 중요도와 백업 수준에 따라 OS 단의 **물리적 백업(전체 백업)**과 Export 유틸리티 등을 이용한 **논리적 백업**을 적절히 혼용하여 운영해야 합니다.
