### 1. 데이터베이스 성능 개선 개요 및 분석
데이터베이스 성능을 평가하는 기준과 성능 저하의 주원인을 파악하고, 이를 분석하는 방법에 대해 설명합니다.

*   **성능 평가 지표:** 데이터베이스 성능은 주어진 시간에 얼마나 많은 작업을 처리하는지를 나타내는 **처리량(Throughput)**과 화면에 결과를 얼마나 빨리 보여주는지를 뜻하는 **응답 속도(Response Time)**로 나뉩니다. 대량의 배치 작업은 처리량이 중요하고, 온라인 웹 서비스는 응답 속도가 중요합니다.
*   **성능 저하의 핵심 원인:** 성능 문제의 90% 이상은 CPU나 메모리 부족이 아닌, 비효율적인 DB I/O(디스크 입출력)에서 발생합니다. 수만 블록을 읽어야 하는 비효율적인 SQL을 하드웨어 증설로 튜닝하는 것은 한계가 있습니다.
*   **성능 튜닝의 접근 순서:** 바깥(하드웨어)에서부터 접근하기보다는, **SQL 튜닝 -> DB 오브젝트(인덱스 등) 튜닝 -> DB 인스턴스(메모리/프로세스) 튜닝 -> 하드웨어/OS 튜닝** 순서로 접근하는 것이 가장 바람직합니다.
*   **SQL 트레이스(Trace) 분석:** TKPROF 등의 유틸리티를 통해 SQL의 파싱(Parsing), 실행(Execution), 패치(Fetch) 과정을 분석합니다.
    *   매번 SQL 문장을 새로 조립하여 실행하는 **동적(Dynamic) SQL은 반복적인 하드 파싱 부하를 일으키므로 심각한 성능 저하**를 초래합니다. 
    *   바인딩 변수를 사용하여 한 번 파싱된 실행 계획을 재사용하게 하는 **정적(Static) SQL 작성이 필수적**입니다.

### 2. 조인(Join) 메커니즘의 이해
조인을 회피하고 반정규화를 무리하게 하는 것보다, 상황에 맞는 조인 방식을 적절히 선택하는 것이 물리적 I/O를 줄여 퍼포먼스에 유리함을 설명합니다.

*   **중첩 루프 조인 (Nested Loop Join):** 선행(Driving) 집합에서 후행 집합으로 인덱스를 이용해 건건이 탐침하는 방식입니다. OLTP처럼 데이터량이 적고 빠른 응답이 필요한 부분 범위 처리에 적합하며, 드라이빙 테이블의 순서와 인덱스의 유무가 성능을 좌우합니다.
*   **소트 머지 조인 (Sort Merge Join):** 조인할 두 집합을 연결 고리 컬럼으로 각각 정렬(Sort)한 뒤 병합(Merge)하는 방식입니다. 인덱스가 없어도 사용할 수 있으나 정렬에 따른 부하(디스크 스와핑 등)가 발생할 수 있습니다.
*   **해시 조인 (Hash Join):** 작은 집합을 메모리상에 해시 맵(Build Input)으로 구성한 뒤, 큰 집합(Probe Input)을 읽어가며 해시 함수를 통해 연결하는 방식입니다. 소트 부하가 없기 때문에 대용량 데이터들의 조인(배치 처리 등)에서 탁월한 성능을 발휘합니다.
*   **특수 조인 기법:** 차원(Dimension) 테이블이 작을 때 이들의 카테시안 곱을 먼저 만든 뒤 대용량 팩트 테이블과 한 번만 조인하는 **스타 조인(Star Join)**, 랜덤 I/O를 줄이기 위해 로우 아이디(Row ID)를 정렬하여 탐침하는 **하이브리드 조인** 등이 있습니다.

### 3. 애플리케이션 성능 개선 (온라인 vs 배치)
개발된 프로그램의 성격(온라인 서비스 vs 대량 배치 작업)에 따라 튜닝의 목표와 접근법이 완전히 달라져야 함을 강조합니다.

*   **온라인 프로그램 튜닝:** 
    *   **응답 속도 최적화:** 전체 데이터를 다 읽고 정렬하기보다는, 인덱스를 활용하여 원하는 조건의 데이터를 읽는 즉시 응답할 수 있는 **부분 범위 처리(Partial Range Scan)** 방식을 적극 활용해야 합니다.
    *   게시판이나 조회 화면에서 불필요한 함수 사용은 문맥 전환(Context Switching)과 내부 DB 콜을 연쇄적으로 발생시켜 부하의 주범이 되므로 주의해야 합니다.
*   **배치 프로그램 튜닝:**
    *   **처리량 최적화:** 500만 건을 건건이 읽어서 루프(Loop)를 도는 절차적 프로그래밍(마치 삽으로 건물을 짓는 행위)을 지양해야 합니다.
    *   대신, 집합적 처리 방식을 도입하여 해시 조인을 쓰거나 한 번에 수천 건을 배열로 처리(Array Processing)하는 등, **포크레인 방식의 집합적 SQL**로 변경하여 DB 콜을 최소화해야 합니다.
*   **분석 함수 (Analytic Functions) 활용:** 누적 합계(Running Total), 랭킹(Ranking), 이전/다음 행 참조(Lag/Lead) 등 복잡한 절차적 로직이 필요한 계산을 SQL 단일 문장으로 빠르게 처리할 수 있게 해줍니다. 
    *   서브 토탈(소계)을 자동으로 만들어 주는 **Rollup, Cube**, 혹은 사용자가 원하는 조합만 명시하는 **Grouping Sets** 등을 활용해 다차원 분석 쿼리를 효율화할 수 있습니다.

### 4. 스토리지, 오브젝트 및 인스턴스 튜닝
SQL이나 애플리케이션 최적화 후, 데이터를 물리적으로 담는 구조와 운영체제 자원을 튜닝하는 방법입니다.

*   **파티셔닝 (Partitioning):** 수억 건의 대용량 테이블을 작은 단위의 파티션으로 분할하여 저장합니다. 이를 통해 특정 파티션만 풀 스캔(Full Scan)하여 I/O를 대폭 줄이고, 병렬 처리(Parallel DML)나 관리적 측면(백업/복구)의 가용성을 높입니다.
*   **오브젝트 튜닝:** 테이블과 인덱스의 스토리지(디스크)를 분리하여 경합을 줄이고, 업데이트가 잦아 불균형해진 스큐(Skew) 인덱스를 주기적으로 재구성해줍니다.
*   **인스턴스 및 래치 튜닝:** 버퍼 캐시나 해시 메모리 사이즈를 조정하여 물리적 디스크 I/O를 최소화합니다. OLTP 환경에서 다수의 동시 접속으로 인한 잠금(Locking)/래치(Latch) 경합이 발생하지 않도록 트랜잭션 옵션이나 프리리스트(Freelist) 여유 공간을 넉넉히 설정해줍니다.
